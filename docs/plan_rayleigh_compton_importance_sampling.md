サンプリングを逆変換2段階方式に置き換え、反復回数を最悪ケースで約4,870倍削減
（cumtimeは約46倍、chest_room.yaml全体では約1.70倍）。EGS5相互検証3ケース

[docs/plan_transport_speedup.md](plan_transport_speedup.md) Phase 2で「機械的ベクトル化では
解決不能、アルゴリズム再設計が必要」と判断してスコープ外に申し送った本命の高速化。
高速化計画の設計判断1（数値ロジックは変えない）を**意図的に超える**計画であり、
物理モデルのサンプリング手順そのものを変えるため、統計的同等性の検証とEGS5相互検証の

この計画書は設計判断と実施順序を事前に固定してある。実行者は各Phaseを上から順に
実施し、**各Phaseの受入基準を通してから次へ進む**こと。判断に迷う箇所は勝手に設計
変更せず「設計判断（確定事項）」に戻るか、ユーザーに確認する。

## 背景と目的

### 前計画からの申し送り事項（出発点）

`plan_transport_speedup.md` Phase 2実施記録（2026-07-20）で実測により確定した事実:

- `sample_rayleigh_cos_theta`（[chatcarlo/physics.py](../chatcarlo/physics.py)）の棄却ループは
  **受理率そのものが低い**（水80 keV・n=20,000のバッチで反復約5,499回）。原因は包絡線
  `2Z²` が緩すぎること — 特に軽元素（H等）はF(Z,q)が前方に強く偏るため、一様提案の
  cosθが中〜大角度に外れると受理率が数%以下に落ちる。診断領域の高エネルギー端
  （100〜150 keV）ほど悪化する。
- 棄却ループは後半 `pending` が1〜数個まで縮む長い裾を持ち、探索の共通化などの
  機械的ベクトル化は**実測で悪化**した（不採用・revert済み）。同じ轍を踏まないこと。
- Phase 3（マルチプロセス並列化）は完了済みだが、**単体ワーカーのアルゴリズム効率は
  未改善**。n=1e8級の遮蔽評価統計を狙うにはこの再設計が次の一手（[[future_directions]]）。

### 目的

反復数千回の棄却ループを、**受理率が数学的に50%以上保証される2段階方式**に置き換え、
単コアのサンプリングコストを桁で削減する。目標分布（微分断面積）は一切変えない —
変えるのは同じ分布から抽選する**手順**だけである。この「分布は同じ・手順が違う」を
証明することが本計画の検証の全てである。

### 目標（事前登録）

- **主目標（算法）**: レイリー角度サンプリングの棄却反復回数を、最悪ケース
  （水・soft_tissue 80〜150 keV）で **数千回→10回以下**（理論期待値は2回以下）。
- **主目標（速度）**: Phase 0で再取得する現行プロファイルを基準に、`chest_room.yaml`
  n=1e5 単コアで、`sample_rayleigh_cos_theta`＋`sample_compton_bound` の合計時間を
  **1/5以下**にする。全体壁時間の改善率はレイリー/コンプトンの寄与率に依存するため
  目標にしない（Phase 0で寄与率を記録し、達成可能な上限をそこで確定させる）。
- **絶対制約**: 物理結果の統計的同等性。§検証プロトコルの3層（分布同等・回帰テスト・
  EGS5相互検証）を全て通す。1つでも落ちたら不採用・revert。

## アルゴリズム設計（数学的準備）

### レイリー: 現行と提案の対比

目標分布（両者で不変）:

```
dσ/dcosθ ∝ (1 + cos²θ) · F(Z, q)²,   q = E·sin(θ/2)/hc  [Å⁻¹]
```

**現行**: cosθ ∈ [-1,1] を一様提案し、`(1+cos²θ)F²/(2Z²)` で棄却。
F(Z,q)はq=0（前方）で最大値Zを取り急減するため、包絡線2Z²はほぼ全域で過大 → 低受理率。

**提案（2段階方式）**: 変数変換 `x ≡ q²` を使う。恒等式 `sin²(θ/2) = (1-cosθ)/2` より

```
x = q² = (E/hc)² · (1 - cosθ)/2,   x_max(E) = (E/hc)²  （θ=πで達成）
```

**cosθはxの1次式**（`cosθ = 1 - 2x/x_max`）なのでヤコビアン `|dcosθ/dx| = 2/x_max` は
**定数**。したがって目標分布はxの空間で

```
p(x) ∝ (1 + cos²θ(x)) · F(Z, √x)²,   x ∈ [0, x_max(E)]
```

と書け、これを次の2段に分解できる:

1. **逆変換サンプリング**: 累積分布 `A(x) = ∫₀ˣ F(Z,√x')² dx'` を元素Zごとに事前
   テーブル化しておき、`ξ₁ ~ U(0, A(x_max(E)))` から `x = A⁻¹(ξ₁)` を引く
   （searchsorted＋区間内解析反転、1回で決まる）。
2. **角度因子の棄却**: `ξ₂ ≤ (1 + cos²θ(x))/2` で受理。この因子は [1/2, 1] に
   収まるため**受理率は50%以上が数学的に保証**され、期待試行回数は2回以下。

これは**EGS5自身が採用している標準手法と同型**である（EGS5マニュアルのcoherent
scattering節: A(x)テーブルからの逆変換＋(1+cos²θ)/2棄却。EGSnrc PIRS-701も同方式）。
つまり「独自の工夫」ではなく、相互検証の相手と同じ土俵に乗る変更であり、EGS5との
一致はむしろ改善方向にしか働かない。

**テーブル化の中身**: 既存の `rayleigh_form_factor_table(z)`（materials.py、q∈[0,20]Å⁻¹・
2000点）を土台に、x=q²格子上でF²を台形積分した累積 `A(x)` を元素Zごとに構築
（`@lru_cache`、既存テーブル群と同じパターン）。逆引きは「A格子をsearchsorted →
区間内はF²を区間定数（または1次）とみなして解析的に反転」。区間内近似が分布に入れる
歪みは検証プロトコル(a)で数値的に上限確認する。

### コンプトン: 現行手法の位置づけと本計画でのスコープ

現行の `sample_compton_bound`（Kahn型KN提案 → S(Z,q)/Z追加棄却）は、**EGS5の束縛
コンプトン（incoh=1）と同型の標準手法**であり、レイリーのような「包絡線が桁で緩い」
構造的欠陥はない。受理率の損失は (KN Kahn受理率) × (S/Z受理率 ≈ σ_bound/σ_KN) で、
診断領域の水では後者は1に近い。

したがって本計画では**コンプトンは「計測して判断」に留める**（Phase 2）。Phase 0の
プロファイルで `sample_compton_bound` の反復回数・時間寄与を計測し、レイリー改善後も
なお支配的な場合に限り、対策を設計してユーザーに提案する（勝手に着手しない）。
候補は S(Z,q)/Z ≈ 1 とみなせる領域（q大・低Z）での棄却スキップ等の軽微な最適化で、
レイリーのような全面再設計は現時点で根拠がない。

## 設計判断（確定事項）

1. **目標分布は変えない。サンプリング手順だけ変える**。微分断面積・断面積テーブル・
   元素抽選（`sample_rayleigh_element`）・化合物の扱いは一切不変。変更は
   `sample_rayleigh_cos_theta` の内部実装と、materials.py への累積テーブル追加のみ。
   公開関数のシグネチャと戻り値の意味は不変に保つ。

2. **乱数消費列は変わる（明示的に許容する）**。試行回数が変わるため同一seedビット一致は
   新旧で成立しない。これは高速化計画の設計判断4で既に確立した「統計的同等＋誤差上限」
   基準に載せる。`tests/test_reproducibility.py` は**新実装同士の再現性**（同一seed→
   同一結果）を引き続き保証する（新実装に切り替えた後のビット一致は要求し続ける）。

3. **旧実装は削除せず検証用に残す**。`_sample_rayleigh_cos_theta_uniform`（旧・一様提案版）
   として private 関数で保持し、新旧サンプラーの2標本検定（検証プロトコル(a)）を恒久的な
   回帰テストにする。輸送本体からの呼び出しは新実装のみ。

4. **元素グループ化は既存の教訓に従う**。Zごとのテーブル参照ループは `set()` 直接反復
   ではなくソート済み順で回す（[[project_status]]の「set反復順で再現性が静かに壊れた」
   教訓。materials.py `material_groups` と同じ理由）。

5. **q_max(E)がテーブル上限を超えないことを構築時に検証する**。x_max = (E/hc)² は
   E=150 keVで q≈12.1 Å⁻¹（テーブル上限20 Å⁻¹に対し余裕あり — 既存テーブルの
   docstringに記載済み）。エネルギー上限150 keVは高速化計画の事後レビューで
   scene.py にfail-fast検証済みなので、輸送中に超えることはないが、累積テーブル
   構築時に assert で明文化する。

6. **検証は3層全部やる。ショートカットしない**。Phase 1検証（2026-07-20）で「単純な
   値の一致でなく統計的同等性の確認が必要」が実例で裏付けられている。本計画は
   サンプリング手順自体を変えるため、なおさら分布レベルの検証（下記(a)）を最初に置く。

## 検証プロトコル（全Phase共通の受入基準）

- **(a) 分布同等性（新規・本計画の核心）**: 代表点マトリクス
  Z ∈ {1(H), 8(O), 20(Ca), 82(Pb)} × E ∈ {20, 60, 80, 150} keV の全16組について、
  各n=1e6サンプルで
  - 新旧サンプラーの**2標本KS検定**（有意水準は多重比較補正込みで事前固定: 16組
    に対しBonferroni補正でα=0.05/16）
  - 目標pdf（(1+cos²θ)F²を数値積分で規格化した理論分布）に対する新実装の
    **1標本適合度検定＋モーメント比較**（⟨cosθ⟩, ⟨cos²θ⟩ の理論値との差が
    標準誤差の3倍以内）
  を通す。恒久テスト化する際は計算時間を抑えるため縮小版（n=1e5・代表4組）を
  `tests/` に置き、フル版はスクリプトとして `docs/` 配下に残す。
- **(b) 既存回帰網**: `pytest tests/ -q` 全通過（Phase 3完了時点で全テスト、
  test_xs_tables.py 74件・test_reproducibility.py 含む）。
- **(c) EGS5相互検証の再実行**: 代表ケースで合否が変わらないこと。ケース選定は
  レイリー感度を考慮して以下の3件（[docs/plan_egs5_crosscheck.md](plan_egs5_crosscheck.md)
  の既存プロトコル・既存スクリプトを再利用）:
  - 水60 keV 一次透過率（`egs5_crosscheck/run_chatcarlo_water60.py`）— 基本の安全網
  - PDD60（`egs5_crosscheck/run_chatcarlo_pdd60.py`）— 散乱線の空間分布に感度
  - BSF60（`egs5_crosscheck/run_chatcarlo_bsf60.py`）— 後方散乱はレイリー/コンプトンの
    角度分布の歪みに最も敏感
  比較は統計的同等性（SEM考慮）で判定する。正典JSONを上書きしない
  （Phase 1事後レビューでの一時上書き事故の教訓 — 必ず別ファイルに出力して比較）。

### Phase 1: レイリー逆変換2段階方式の実装（本丸）

- materials.py: `rayleigh_cumulative_table(z)` を追加 — x=q²格子上の累積
  `A(x) = ∫F²dx'`。既存 `rayleigh_form_factor_table` と同じキャッシュ・命名パターン。
  格子はF²の変化が速いx小域に密になるよう既存q格子（線形）をx=q²に写して使う
  （x空間では自動的に前方が密になる）。設計判断5のassertをここに置く。
- physics.py: `sample_rayleigh_cos_theta` の内部を2段階方式に差し替え。
  旧実装は `_sample_rayleigh_cos_theta_uniform` として保持（設計判断3）。
  Zごとのループはソート順（設計判断4）。ξ₁（逆変換）とξ₂（角度因子棄却）の
  消費順をdocstringに明記する。
- 棄却ループの構造は現行と同じ「pendingを縮めるwhile」でよい — 受理率≥50%なので
  裾は数回で消え、前計画で問題になった「長い裾のオーバーヘッド」自体が消滅する。
- 受入基準: 検証プロトコル(a)(b) ＋ 反復回数の実測（主目標: 最悪ケースで10回以下）
  ＋ プロファイル再取得でレイリー角度サンプリングの時間が支配項から消えること。

## リスクと落とし穴

- **累積テーブルの区間内近似バイアス（最重要）**: 逆変換の区間内反転でF²を区間定数/
  1次とみなす近似が、そのまま抽選分布の歪みになる（棄却法と違い逆変換は「テーブルが
  分布そのもの」）。検証プロトコル(a)の適合度検定・モーメント比較で数値的に上限確認し、
  足りなければ格子を密にする。**ここをスキップして速度だけ確認するのは禁止**。
- **q_max境界の扱い**: `A(x_max(E))` での切断を忘れると、テーブル全域からxを引いて
  しまいcosθ < -1の非物理値を生む。cosθ計算後の `np.clip` で隠蔽せず、切断を明示的に
  実装してテストする（clipは丸め誤差対策のみに限定）。
- **乱数消費列の変更**: 設計判断2で織り込み済み。新旧のビット一致を期待するテストが
  もし残っていれば、このPhaseで「新実装同士の再現性」に書き換える。
- **機械的ベクトル化の再発**: 前計画Phase 2で不採用になった「複数元素まとめ探索」を
  この機会に再導入しない。受理率が上がれば裾問題は消えるので、追加の共通化は不要。
- **旧実装の削除誘惑**: 設計判断3のとおり検証用に残す。「使われていないから削除」を
  後続セッションがやらないよう、docstringに保持理由を明記する。
- **正典データの上書き**: EGS5再検証で正典JSON/ログを上書きしない（前例あり）。
  比較出力は必ず別名ファイルへ。

## 本計画のスコープ外（申し送り）

- **分散低減（粒子スプリッティング/importance sampling/Russian roulette)**: 遮蔽背後の
  ヒットゼロ問題の本筋対策だが、乱数構造・統計重みの導入を伴う別軸の計画
  （前計画Phase 4の申し送りを維持）。
- **コンプトンの全面再設計**: Phase 2の計測で根拠が出ない限りやらない。
- **numba JIT化**: 本計画でxraylib非依存の純テーブル演算になるため技術的には可能に
  なるが、依存追加の判断は別途（前計画Phase 3(b)の位置づけを維持）。

## 実施記録

### Phase 0（2026-07-21、完了）

`chest_room.yaml` n=1e5, seed=42のプロファイル再取得（着手前・前計画Phase1+3完了
時点）。総壁時間1.774秒中、`sample_rayleigh_cos_theta`がcumtime 0.698秒（**41.7%**）
で最大の単一ボトルネック、`sample_compton_bound`は0.240秒（14.3%）。詳細は
[docs/speedup_baseline/RCI_PHASE0_BASELINE.md](speedup_baseline/RCI_PHASE0_BASELINE.md)。

棄却ループの反復回数を代表16組で実測: レイリー（現行cosθ一様提案）は最悪ケース
Z=1(H)・E=150keVで**平均4,872試行/光子**（Z=8・O系でも885試行）。コンプトンは
全16組で平均3.2試行以下——既に高受理率で構造的欠陥なし、想定通り。

検証ハーネス[verify_rayleigh_distribution.py](rci_verification/verify_rayleigh_distribution.py)
を作成する過程で、**当初計画のn=1,000,000（新旧とも同一）は非現実的**と判明した
（旧実装がZ=1・150keVでn=100,000だけで390秒——n=1e6なら1組で約65分）。旧実装は
本計画で置き換える対象そのもので高精度検証の必要性も薄いため、**サンプル数を
新旧非対称に設計変更**: 新実装n=1,000,000・旧実装n=20,000（`ks_2samp`は不等
サンプルサイズに対応）。この変更はPhase 1実装後にハーネスとして一体的に検証済み
（下記Phase 1参照）。

### Phase 1（2026-07-21、完了・主目標を大幅に上回って達成）

`materials.py`に`rayleigh_cumulative_table(z)`を追加（x=q²空間でのF²累積積分
A(x)）、`physics.py`の`sample_rayleigh_cos_theta`を2段階方式（逆変換+角度因子
棄却）に差し替え。旧実装は`_sample_rayleigh_cos_theta_uniform`として保持。

**検証プロトコル(a)**: 全16組で新実装vs理論pdf（一標本KS検定、Bonferroni補正
α=0.003125）**全通過**（p値0.0295〜0.9593）、旧実装vs理論pdfも全通過（ハーネス
較正、Phase 0の目的も同時達成）、新実装vs旧実装（2標本KS検定）**全通過**
（p値0.0656〜0.9595）。実行時間**19秒**（当初計画のn=1e6均一設計なら旧実装だけで
数時間かかっていたはずの検証がこの時間で完了した点自体が本計画の効果を裏付ける）。
モーメント（⟨cosθ⟩,⟨cos²θ⟩）も全16組で理論値・新旧間とも良く一致。

**検証プロトコル(b)**: 既存回帰網`pytest tests/ -q` 234件全通過（新規追加前）。
恒久回帰テストとして`tests/test_rayleigh_distribution.py`（代表4組・分布同等性、
n_new=100,000/n_old=5,000で数秒完走）と`tests/test_rayleigh.py`への3件追加
（境界値clip確認、テーブル範囲外エネルギーのfail-fast確認）を追加し、
最終的に**245件全通過**。

**セルフレビューでの改善（2026-07-21、事後追記、最終246件全通過）**:
- **範囲外検出を`assert`→`ValueError`に変更し、上限をテーブルから直接導出**:
  当初は設計判断5に沿って`assert`で実装したが、`assert`は`python -O`で無効化
  されるため安全不変条件の担保には不適で、前計画`plan_transport_speedup.md`
  事後レビューの「範囲外エネルギーのfail-lateクラッシュ」教訓を繰り返す恐れが
  あった。materials.pyの`_element_interp_index_frac`が既に採っている`ValueError`
  パターンに揃えた（`tests/test_rayleigh.py`の該当テストも`pytest.raises(ValueError)`
  に更新）。さらに当初は上限を`_RAYLEIGH_Q_MAX_ANGSTROM=20.0`という**materials.py
  側のq_max既定を別に写した定数**でチェックしていたが、これは両者が将来divergeすると
  「チェックは通るのにテーブルは実際には狭く、`np.interp`が静かに端値クランプして
  後方散乱が欠落する」潜在バグになる。定数を廃し、ループ内で**実テーブルの端
  `x_grid[-1]`を直接**上限に使う形に変えて結合を解消した。
- **混在元素配列の分布テストを追加**: 既存の分布テストは単一元素配列
  （`np.full(n, z)`）のみで、実輸送で通る「複数元素混在配列を`for z in
  sorted(set(...))`でグループ化しマスク振り分けする」経路が間接的にしか
  検証されていなかった。O/Pb交互配列で各元素の部分集合が自元素の理論分布に
  適合することを直接確認するテストを`test_rayleigh_distribution.py`に追加。

**反復回数（主目標: 最悪ケースで10回以下、理論期待値2回以下）**: 実測は
**全16組で平均1.00〜1.37試行**——理論上界の2回すら下回った（前方散乱への
重みが元々強いため、角度因子の平均受理率が50%よりかなり高いケースが大半）。
最悪ケースだった旧実装のZ=1・E=150keV（平均4,872試行）は新実装で平均1.0006
試行、**約4,870倍の反復削減**。

**速度（chest_room.yaml n=1e5）**: 1.774秒→1.046秒（**約1.70倍**）。
`sample_rayleigh_cos_theta`のcumtimeは0.698秒→0.015秒（**約46倍**）。ただし
事前登録した副次目標「レイリー+コンプトン合計を1/5以下に」は**未達**
（0.938秒→0.250秒、約3.75倍で目安の5倍に届かず）——これはコンプトン側
（0.240秒→0.235秒、意図的に無改修）が実質不変のまま、レイリーが消えたことで
「合計」に占める相対シェアが上がったことによる見かけ上の未達であり、
レイリー単体の反復回数削減という主目標に対しては大幅に上回って達成している。
この副次目標の立て方自体が「両方縮む」前提の設計ミスだったと認め、後述の
Phase 2判断に反映する。

### Phase 2（2026-07-21、完了・対策不要と判断）

Phase 1後のプロファイル（chest_room.yaml n=1e5）で`sample_compton_bound`の
残存寄与を計測: cumtime 0.219秒 / `transport_photons`全体0.882秒 = **約24.8%**
（Phase 0時点の14.3%から相対的に上昇——上記の通りレイリーが消えた分の相対効果）。
計画書の目安「10%以上」を数値上は超えている。

Phase 0で実測済みの反復回数（全16組で平均3.2試行以下、既に高受理率）と合わせて
判断すると、コンプトンの残存コストは**棄却ループの構造的非効率（低受理率）が
原因ではない**——KN Kahn提案自体が自由電子近似で50%以上の高受理率を持ち、
S(Z,q)/Z追加棄却の損失も診断領域では小さい。

**セルフレビューでの内訳精査（2026-07-21、事後追記）**: `sample_compton_bound`
（cumtime 0.219秒）の内訳を tottime で分解したところ、**棄却ループ本体
（`_propose_free_electron_kn`）はtottime 0.005秒でほぼ無償**、コストの実体は
`sample_compton_element`（cumtime 0.166秒＝`compton_element_weights`経由の
断面積テーブル引き）だった。つまりコンプトンの残存コストは「棄却サンプリング
問題」ですらなく、**レイリー消滅後に残ったもう一つのボトルネック
（断面積テーブル引き、`_element_interp_index_frac`/`searchsorted`系）と同一
カテゴリ**である。この事実は「本計画（棄却サンプリング再設計）のスコープでは
対策不要」という結論を一段と強くする——コンプトンには逆変換方式を適用できる
数学的な足がかりがないどころか、そもそも棄却ループが遅いわけではない。

**判断: 対策不要、Phase 2は計測記録のみで完了とする**（実装は行わない）。
プロファイル上の残りのボトルネックは`_element_interp_index_frac`/`searchsorted`
系（断面積テーブル引き、cumtime 0.433秒）であり、これは前計画
`plan_transport_speedup.md`のスコープ（xraylib呼び出しのテーブル化）に属する
別問題——本計画の対象外。ユーザーが追加の高速化を望む場合は、コンプトンの
軽微な最適化（S/Z≈1領域での棄却スキップ等、計画書のPhase 2記載の候補）を
別途検討可能だが、上記の理由から優先度は高くないと判断し、本計画では実施しない。

### Phase 3（2026-07-21、完了）

検証プロトコル(c)の3ケースを再実行。**正典データは一切上書きしていない**
（水60・BSF60は結果をファイル出力しないスクリプト、PDD60は出力先を正典とは
別名（`chatcarlo_pdd60_results_rci_recheck.json`等）に変更したコピーで実行——
Phase 1事後レビューでの一時上書き事故の教訓を踏まえた対応、`git status`で
正典ファイル無変更を確認済み）。この別名JSONは集計値を本記録に転記した後、
再生成可能な一時成果物としてリポジトリからは削除した（数値の根拠は本記録と
`run_chatcarlo_pdd60.py`＋別名化手順で再現できる）。

- **水60keV一次透過率**（`run_chatcarlo_water60.py`、n=500,000）: **12.589%**、
  正典値と完全一致。無散乱透過はレイリー角度サンプリングと無関係
  （n_scatter==0の光子のみが対象）なので当然の結果——基本の安全網が
  無傷であることの確認。
- **BSF60後方散乱係数**（`run_chatcarlo_bsf60_thinslab.py`、n=500,000×2）:
  BSF_w(新実装) = 1.3607 ± 0.0021。旧実装の正典値1.3586±0.0022とは
  0.69σで統計的に一致。EGS5値1.3651±0.0058とは**0.71σ・相対0.32%**——
  事前登録基準（2σかつ2%以内）を大きく余裕を持って満たす。後方散乱は
  レイリー/コンプトンの角度分布の歪みに最も敏感な指標であり、ここが
  クリアなのは本計画の核心的な安全網。
- **PDD60深部線量分布**（`run_chatcarlo_pdd60.py`のコピー、n=5,000,000、
  47ビン）: 47ビン平均相対差0.1626%（旧実装の正典値では0.0183%、いずれも
  2%の事前登録閾値に対し十分小さい）。統計誤差<1%の43ビン中、事前登録基準
  （σ<2かつ|相対差|<2%）の合格数は**40/43**（旧実装の正典比較では42/43）。
  1ビンあたり2σ閾値なら偶然だけで平均約2ビンが超えることが期待されるため
  （有意水準5%×43≈2.15、期待値は期待値の線形性で成り立ちビン間独立性は
  不要——隣接PDD/側方ビンは史を共有し相関があるが、相関はむしろ近接ビンの
  同時ニアミスを起こりやすくする方向）、新実装の3/43という不合格数は統計的
  ゆらぎの範囲内であり、体系的な乖離ではないと判断する（旧実装の1/43もこの
  期待値の範囲内）。この判定を最終的に担保するのは下記の別seed符号反転検証。

  **セルフレビューでの追加検証（2026-07-21）**: 新実装の平均相対差が旧実装
  （0.0183%）よりわずかに大きい（0.1626%）ことが系統誤差でないことを、別seed
  （seed=202）でのn=5,000,000再実行で確認した。47ビン平均相対差は
  **+0.1626%（seed=1）→ −0.0333%（seed=202）と符号が反転**——系統バイアスなら
  seedによらず同符号・同程度で再現するはずであり、この振れは乱数実現の
  ゆらぎ（1ビンあたりSEM約0.5%、47ビン平均で約0.07%相当）そのものである。
  合格数も42(旧)→40(seed=1)→39(seed=202)と偶然期待値2.15±Poissonの範囲で
  振れており、置き換えが物理を歪めていないことの追加証拠になった。

**結論**: 3ケースいずれも合否判定が変わらず、レイリー2段階方式への置き換えが
物理結果の統計的同等性を壊していないことを確認した。

**セルフレビューで検出・修正した事故（2026-07-21）**: 上記の第2seed検証の際、
`compare_pdd60_incoh1.py`が**出力先`comparison_table.csv`（tracked）を絶対パスで
ハードコードしている**ことを見落とし、比較スクリプトのコピーを複数回実行した
ことでこの正典的な比較結果CSVを新実装データで上書きしてしまった（正典JSON
`chatcarlo_pdd60_results.json`自体は無傷）。`git status`でtracked fileの変更に
気づき`git checkout`で復元、復元後CSVが正典JSONとbyte一致することを確認済み。
**教訓**: `egs5_crosscheck/`の比較スクリプト群は出力パスをハードコードして
いるものがあり、コピーして別データで走らせるとtracked fileを黙って書き換える。
今後この種のスクリプトを流用する際は出力パスもリダイレクト対象に含め、実行後は
必ず`git status`で正典ファイルの無変更を確認すること（前計画Phase1事後レビューの
「正典JSON一時上書き事故」と同型の落とし穴が、比較スクリプト側にも潜んでいた）。

### 総合結果（2026-07-21時点）

`chest_room.yaml` n=1e5: 1.774秒→1.046秒（**約1.70倍**）。前計画Phase1+3
（3.104秒→1.774秒、約1.75倍）と合わせ、着手前の3.104秒から**通算約2.97倍**。
レイリー角度サンプリング単体は反復回数で約4,870倍（最悪ケース）、
cumtimeで約46倍——本計画の主目標（アルゴリズム再設計による桁の改善）は
達成。「レイリー+コンプトン合計を1/5以下」という副次的な速度目標は
未達（実測3.75倍）だが、これはコンプトンを意図的に無改修としたことによる
相対シェアの見かけ上の結果であり、Phase 2判断（対策不要）と整合する。

残るボトルネックは断面積テーブル引き（`_element_interp_index_frac`/
`searchsorted`系）で、これは前計画`plan_transport_speedup.md`のスコープに
属する別問題。より大きなスケール（n=1e8等）を狙う場合の次の一手は、
このテーブル引き自体の最適化（例: 3種の断面積だけでなくレイリー/コンプトン
テーブルの検索も含めた探索の一本化）または並列度のさらなる向上であり、
いずれも本計画のスコープ外——ユーザー判断で別計画を起こすこと。
