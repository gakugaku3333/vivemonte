対象: 学生向け視覚教材として、MC輸送の飛跡ログを時系列再生するアニメーションHTMLを生成する
新コマンド `chatcarlo animate` の実装。コンセプト文書（一人称視点／対向・リレー視点、時間の
デザイン、色・音の演出、誠実さの設計）を、既存コードベースの資産の上に段階実装する。

この計画書は Sonnet クラスのモデルが1 Phaseずつ独立に実行できる粒度で書いてある。
各Phaseの末尾に受け入れ基準があり、それを満たしてから次へ進むこと。

[tests/test_animation.py](../tests/test_animation.py)（Python側11テスト）、

## 0. 現状資産の棚卸し（実装前に必ず読むファイル）

| 資産 | 場所 | アニメーションでの役割 |
|---|---|---|
| `TrajectoryRecorder` | [chatcarlo/trajectory.py](../chatcarlo/trajectory.py) | 飛行区間ごとの start/end/energy/event/photon_id を記録済み。**乱数を消費しない**設計（recorder有無で輸送結果が同一seedでビット一致） |
| `trajectories_to_json` | 同上 | 光子ごとのポリライン `{points, energies, events}` に整形済み |
| recorder呼び出し側 | [chatcarlo/transport.py](../chatcarlo/transport.py) の `transport_photons` 末尾 | event種別（boundary / photoelectric / compton / rayleigh / fluorescence / escape）を確定してrecord。この時点で `mat`（材料名配列）も手元にある |
| 自己完結HTML 3Dビューア | [chatcarlo/preview.py](../chatcarlo/preview.py) | vanilla JS + canvas の透視投影（`project()`）、シーン形状のJSON化（`scene_to_json`）、エネルギー→色相マッピング（`hue = 240*(E/Emax)`、青=高E→赤=低E）が実装済み |
| `trace` CLI | [chatcarlo/__main__.py](../chatcarlo/__main__.py) `cmd_trace` | 小history実行→recorder→HTML出力の配管の実例。animateはこれを雛形にする |

**コンセプト文書のデータ要件（§2）は既にほぼ満たされている。** 不足は以下の3点のみで、
Phase 0で埋める:

1. 区間ごとの**材料名**（HUDで「いまコンクリート内」等を表示するため）
2. 光子ごとの**転帰分類**（無散乱透過／散乱後脱出／吸収／後方散乱、の選択メニュー用）
3. 進行方向・散乱角・波長 — これらは points/energies から**JS側で導出**でき、データ追加は不要
   （方向 = 隣接点の差、散乱角 = 隣接方向の内積、λ[pm] = 1239.84 / E[keV]）

終端理由について: transport.py を確認済みで、光子の終わり方は
**「光電吸収（蛍光放出なし）」か「脱出」の2つだけ**。エネルギーカットオフは存在しない
（蛍光線が5 keV未満の場合も event は "photoelectric" のまま局所吸収）。分類ロジックは
この2値を前提としてよい。

## 1. 設計判断（コンセプト文書からの差分と理由）

### 1.1 three.js は使わない（vanilla JS + canvas を継続）

コンセプト文書§8は three.js を挙げているが、本プロジェクトの preview.py は
「外部ライブラリ・CDNに依存しない自己完結HTML」を設計原則としており（承認ゲートで
どこでも開けることが価値）、アニメーションHTMLも同じ原則に従う。three.jsを
インライン埋め込みすると1ファイル1MB超になり、バージョン固定・ライセンス表記の
管理も増える。必要なのは「位置と姿勢を持つ透視カメラ」1つであり、これは
preview.pyの `project()`（注視点周回カメラ）を一般化した約80行の数学で足りる
（§Phase 1 に実装レシピを記載）。

### 1.2 新コマンド `animate`・新モジュール `chatcarlo/animate.py`

既存 `trace`（俯瞰・静止画的な全軌跡表示）はデバッグ・承認ゲート用として**一切変更しない**。
アニメーションは別テンプレートを持つ `chatcarlo/animate.py` に実装し、シーン形状の
JSON化は `preview.scene_to_json` を再利用する。

### 1.3 実装順序: 再生エンジン → 対向・リレー視点 → 一人称視点

コンセプト文書§3.3の教育順序（先に幾何理解、後に体感）と同じ順で実装する。
対向・リレー視点はカメラ姿勢が区間内固定で補間不要（§3.2「実装負荷が低い」）、
一人称はslerp・酔い対策が必要で最後に回す。その前段として、まず既存の
周回カメラのまま「時間再生」だけを作る（Phase 1）。時間デザイン（圧縮・スローモー）は
カメラと独立した関心事なので、カメラ実装前に単体で完成・検証できる。

### 1.4 触ってはいけないもの（全Phase共通の禁止事項）

- **輸送の乱数消費を変えない。** recorder拡張は「記録内容を増やす」だけにする。
  変更後に `tests/` の既存テストが全て通ること（特にwarm-up不変性・再現性系）。
- 既存 `trace` コマンドの出力・挙動を変えない（trajectories_to_jsonにキーを
  **追加**するのは可。traceテンプレートは未知キーを無視する）。
- CDN・外部フォント・外部画像への参照をHTMLに入れない。
- 物理量（断面積・散乱角分布）をJS側で再計算・改変しない。JSは記録された
  ログの**再生装置**に徹する（コンセプト§7の誠実さの根拠）。

## Phase 0: データ整備（Python側、~半日）

### 実装

1. **材料名の記録**: `TrajectoryRecorder.record` に `material` 引数を追加し、
   transport.py の呼び出し（`recorder.record(idx, o, ends, e, event)`）に
   `mat`（その反復の材料名配列、`geometry.material_at(o)` の戻り値）を渡す。
   `trajectories_to_json` の各軌跡に `"materials": [...]`（区間ごと、energiesと同長）を追加。

2. **転帰分類**: trajectory.py に純関数を追加:

   ```python
   def classify_trajectory(traj: dict) -> dict:
       """転帰分類。trajectories_to_json の1要素を受け取り、以下を返す:
       - n_interactions: photoelectric/compton/rayleigh/fluorescence の件数
       - fate: "absorbed"（最終eventがphotoelectric）| "escaped"
       - backscattered: 最終区間方向と初期方向の内積が負ならTrue
       - category: "direct"（n_interactions==0で脱出）| "scattered_escape"
                   | "backscatter"（脱出かつbackscattered）| "absorbed"
       """
   ```

   分類は `trajectories_to_json` の戻り値に `"summary"` キーとして各軌跡へ埋め込む
   （呼び出し側で後付けでもよいが、animate.py と将来の対話AI導線が同じ分類を
   参照するので一箇所で持つ）。

3. **蛍光の扱いに注意**: event "fluorescence" の区間の**次の区間**は等方方向へ
   不連続に方向転換する（光電吸収→蛍光線再放出）。これは散乱角の物理
   （コンプトンの運動学）とは別物なので、分類・後段のHUD表示では
   「散乱角」ではなく「再放出」として区別すること。

### テスト（`tests/test_animation.py` 新規）

- 手組みのrecorderデータ（2〜3光子: 無散乱透過、コンプトン2回→光電、後方散乱脱出）で
  `classify_trajectory` の全カテゴリを検証。
- `trajectories_to_json` に materials が energies と同長で入ることを検証。
- 実シーン（`examples/chest_room.yaml`, n=50, seed固定）で `trace` を実行し、
  既存出力（points/energies/events）が**変更前とバイト一致**することを確認する
  回帰テスト（recorder拡張が輸送に影響しないことの担保）。

### 受け入れ基準

- `.venv/bin/python -m pytest tests/ -q` 全緑（既存テスト含む）。
- `chatcarlo trace examples/chest_room.yaml -n 50 --seed 42` が変更前と同じHTMLを出す。

## Phase 1: `animate` コマンドと再生エンジン（俯瞰カメラ、~1日）

### CLI

```bash
.venv/bin/python -m chatcarlo animate examples/chest_room.yaml -n 300 --seed 42 -o anim.html
```

`cmd_trace` を雛形に `cmd_animate` を追加。輸送・recorder部分は同一、出力だけ
`animate.write_html`。実行後、ターミナルに転帰分類の集計を表示する
（例: `直進透過 212 / 散乱後脱出 61 / 吸収 24 / 後方散乱 3`）— これが将来の
対話AI導線（「後方散乱した光子を見てみましょうか」）の情報源になる。

### HTML構成（`chatcarlo/animate.py` の `_TEMPLATE`）

preview.pyのテンプレートを出発点にし、以下を追加:

1. **光子選択パネル**: 転帰カテゴリ別にグルーピングした軌跡リスト
   （「後方散乱 (3)」等の見出し＋各光子の相互作用回数・終端エネルギー）。
   選択で再生開始。コンセプト§6の「どの光子を追うか選ばせる導線」に相当。

2. **タイムライン設計**（コンセプト§4の時間デザイン）:
   - 実移動時間（フェムト秒）は使わない。区間 i の表示時間を
     `t_i = clamp(0.5 + 2.0 * sqrt(L_i / L_max), 0.6, 2.5)` 秒とする
     （L_i: 区間長。sqrtで長い自由行程を圧縮しつつ長短の順序は保つ）。
   - 相互作用イベント（boundary以外）の直前300msは再生速度0.3×のスローモー、
     イベント時点で400ms停止して相互作用マーカー（traceと同じ記号体系:
     ●光電 ○コンプトン ◇レイリー ■蛍光 ×脱出）をフラッシュ表示。
   - これら定数は先頭の `const TUNE = {...}` に集約（Phase 4で演出強度
     調整UIから触るため）。
   - 実装は「壁時計時間 → 軌跡上の位置」の区分線形写像テーブルを再生開始時に
     構築し、`requestAnimationFrame` で引く方式。スローモーはテーブル構築時に
     該当区間の傾きを変えるだけで表現でき、再生ループ側に分岐が要らない。

3. **描画**: 通過済みの軌跡をエネルギー色（既存の `hue=240*(E/Emax)`）の
   ポリラインで残し、先端に光子の輝点。シーン形状は preview と同じ描画コードを流用。
   カメラはこのPhaseでは既存の周回カメラ（マウス操作可）のまま。

4. **HUD**（コンセプト§5）: 現在エネルギー[keV]・波長[pm]・区間の材料名・
   相互作用回数を数値表示。散乱イベント時に散乱角
   `θ = acos(d_prev·d_next)` を表示（"fluorescence" 直後は「再放出（等方）」と
   表示し散乱角を出さない — Phase 0 §3の注意）。表示ON/OFF切替つき。

5. **再生コントロール**: 再生/一時停止・リスタート・速度（0.5×/1×/2×）・
   光子選択に戻る。

### 検証

ブラウザで `anim.html` を開き（このリポジトリでの検証はBrowserペインで可）、
以下をスクリーンショットで確認:
- 後方散乱カテゴリの光子を選び、方向反転が見えること
- スローモー→停止→マーカーフラッシュの一連が機能すること
- HUDのエネルギーがコンプトン後に減少すること

### 受け入れ基準

- `chatcarlo animate examples/chest_room.yaml -n 300 --seed 42` がHTML生成し、
  ターミナルに転帰集計が出る。
- 全カテゴリの光子が選択・再生でき、上記検証3点が確認できる。
- 生成HTMLに外部URL参照が無い（`grep -c 'https\?://' anim.html` が本文注記以外0）。

## Phase 2: 対向・リレー視点（~1日）

### カメラの数学（vanilla JSでの透視カメラ一般化）

preview.pyの `project()` は「注視点を周回する」カメラなので、位置と姿勢を自由に
持てる形へ一般化する:

```js
// camera = { pos:[x,y,z], right, up, forward }  (正規直交基底)
function projectCam(p, cam) {
  const v = sub(p, cam.pos);
  const x = dot(v, cam.right), y = dot(v, cam.up), z = dot(v, cam.forward);
  if (z < NEAR) return null;                       // 背後・至近はクリップ
  const f = 0.5 * Math.min(w, h) / Math.tan(FOV / 2);   // FOV ~60°
  return [w/2 + f * x / z, h/2 - f * y / z, z];
}
// lookAt(pos, target, upHint=[0,0,1]) から基底を構築（z軸上向きの世界座標）
```

既存の周回カメラも「pos/基底を毎フレーム計算する特殊ケース」としてこの上に
載せ替えると、以降のカメラモード切替が `camera` オブジェクトの差し替えだけになる。

### リレーの仕様（コンセプト§3.2）

- **カメラレッグの構築**: 連続する区間のうち、間のeventが "boundary" で方向が
  不変（隣接方向の内積 > 1−1e−6）のものは1本の「レッグ」に併合する。
  boundaryは材料境界通過であって方向転換ではないため、ここでカットを入れると
  無意味なカメラ切替が多発する。HUDの材料名表示は併合前の区間データから引く。
- レッグ i でのカメラ位置: レッグ終点のさらに先
  `pos = end_i + d_i * offset`（`offset = clamp(0.4 * L_i, 5, 30)` cm）。
  視線は光子の現在位置（= 光子が近づいてくるのを正面から見る）。
  カメラ姿勢はレッグ内ほぼ固定（注視のみ微追従）で、補間処理は不要。
- 光子がレッグ終点（=相互作用点）に達したら、スローモー・停止演出ののち
  次レッグのカメラへ**カット**で切替。入射ベクトルと散乱後ベクトルが
  同一フレームに映る配置になっているか、切替直前のフレームで確認できること
  （これがこの視点の教育目標）。
- UIにカメラモード切替（俯瞰 / 対向リレー）を追加。再生中の切替も許可。

### 受け入れ基準

- コンプトン2回以上の光子で、各相互作用点において「入射してくる軌跡」と
  「散乱して去る軌跡」が1画面に同時に映るスクリーンショットが撮れる。
- boundaryのみで曲がらない区間でカメラが切り替わらない。
- 俯瞰モードが Phase 1 と同一挙動のまま残っている。

## Phase 3: 一人称視点（~1日）

### 仕様（コンセプト§3.1）

- カメラ位置 = 光子位置（微小に後方 `pos - d * 2cm` に置くと先端輝点が視野に残り
  酔いにくい）。姿勢 = 進行方向。
- **方向転換のslerp**: 相互作用での方向転換は、停止演出の400ms＋転換後300msを使い、
  旧方向→新方向を四元数slerpで回す。up ベクトルは世界z軸固定ではなく
  **前区間のupを新forwardに直交射影して引き継ぐ**（平行移動的な引き継ぎ）。
  z軸固定upだと真上/真下向き区間で特異点、大角度転換でロールが暴れる。
- **酔い対策**: FOV 60°固定・ロール抑制（上記up引き継ぎ）・大角度転換
  （θ > 90°）はslerp時間を500msに延長し、画面周辺を軽くフェード。
- 四元数は {slerp, fromLookAt, rotate} の3関数だけ実装すれば足りる（~40行）。

### 終了演出（コンセプト§6の分岐）

- 光電吸収: 1秒かけて暗転、「光電吸収 — 光子はここで消滅した」表示。
- 脱出（透過）: 視界がそのまま開け、明転＋「系外へ脱出（E = xx keV）」。
- 後方散乱は演出不要（一人称では方向転換のslerpそのものが体験になる）。

### 受け入れ基準

- 3視点（俯瞰・対向リレー・一人称）を再生中に相互切替できる。
- 吸収される光子で暗転、透過光子で明転がそれぞれ確認できる。
- 垂直に近い方向へ進む光子（CTシーン等で確認）でカメラが破綻しない。

## Phase 4: 演出の仕上げと誠実さの設計（~半日）

1. **音**（コンセプト§5): WebAudio の OscillatorNode で相互作用時に短いブリップ
   （80ms、周波数 `200 + 1800 * E/E0` Hz — エネルギーが下がるほど低い音）。
   外部音声ファイルは使わない。ブラウザのautoplay制約があるため
   AudioContext は**再生ボタンの初回クリックで生成**する。既定OFF・トグルつき。
2. **注記**（コンセプト§7): 画面隅に常設の小さな注記
   「⚠ 時間・色・カメラの動きは理解のための演出です（軌跡・エネルギー・相互作用は
   計算ログの実データ）」。折りたたみ可・非表示不可。
3. **生データモード**: サイドパネルに現在光子の区間テーブル
   （# / 材料 / E[keV] / 区間長[cm] / event / 散乱角）。再生位置の行をハイライト。
   コンセプト§7の「演出なし表示モード」に相当。
4. **演出強度設定**: TUNE の主要値（スローモー倍率・停止時間・色/音ON/OFF）を
   設定パネルから変更可能に。

### 受け入れ基準

- 注記が全カメラモードで常時視認できる。
- 音OFF既定・ONにした場合のみ発音。HTMLは依然自己完結。
- 生データモードのテーブル値が、Pythonで同一seed・同一光子のrecorderログを
  ダンプした値と一致する（スポットチェックで可）。

## リスクと対策

| リスク | 対策 |
|---|---|
| recorder拡張が輸送結果を変える退行 | Phase 0の回帰テスト（変更前後でtrace出力バイト一致）で担保。record追加引数は記録専用で乱数を触らない |
| 区間数の多い光子（高Z材料内で多重散乱）で再生が長い | タイムライン上限（例: 総再生60秒超なら基準時間を一律スケール）。区間数はn=300規模なら高々数十で、描画負荷は問題にならない |
| 一人称の酔い | FOV固定・up引き継ぎ・大角度時のフェード（Phase 3）。教育導線上も対向リレーを先に見せる前提（§1.3） |
| autoplay制約で音が鳴らない | AudioContextをユーザー操作起点で生成（Phase 4） |
| boundary区間での無駄なカメラカット | レッグ併合（Phase 2）で構造的に排除 |
| 蛍光再放出を「散乱」と誤解させる | HUD・生データモードで「再放出（等方）」と明示（Phase 0/1） |
