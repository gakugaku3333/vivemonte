# 計画: PHITS相互検証パイプライン（scene.yamlを共通言語に）

作成日: 2026-07-13 / ステータス: **計画のみ（未着手）**

**状況（2026-07-15追記）**: 学会抄録（[abstract_krmt2026.md](abstract_krmt2026.md)、
締切2026-07-17 12:00）の結論を「PHITSとの比較検証を行う予定」から実測値付きに
格上げしたく、残り約2日でPhase 0-1（環境構築＋手書き最小ケースの一致確認）まで
トライする。フルパイプライン（Phase 2以降）はこの期間では非現実的なので、
Phase 0（ユーザーのPHITSライセンス取得状況）を最初に確認すること。
間に合わなければ抄録は現行ドラフトのまま提出し、Phase 2以降は通常ペースで継続する。

## 目的

viveMonteの計算結果を確立コードPHITSと自動突き合わせできる体制を作り、
READMEの「教育・研究用」注記を実測比較データ付きで格上げする。
あわせて、以後の新機能（蛍光X線等）を追加するたびにPHITSと自動照合できる
継続的検証網を先に張る。

## 基本方針

「PHITSを万能に操るAI」は目指さない。**viveMonteが表現できる範囲**
（box/sphere/cylinderの3プリミティブ＋X線管光子線源＋吸収線量/H*(10)タリー）
に限定した `scene.yaml → PHITS入力` の機械的変換器を作る。

- 単一の真実源は人間が3Dプレビューで目視確認済みの scene.yaml。
  AIがPHITS入力を自由作文することは**しない**（MCコードの古典的危険＝
  文法的に通るが物理的に間違った入力が、もっともらしい数値を静かに返す
  ことへの対策）
- viveMonte自身を独立推定量として使う。両コードの不一致が入力ミスの
  検出器になる（collision vs track-length 交差検証のコード間版）
- コード選択はPHITS。理由: 日本語ドキュメント充実、研究利用無償（要登録）、
  T-Trackタリー＋線量換算関数でH*(10)が直接出せる、入力1ファイルで完結。
  EGSnrcはジオメトリ記述とH*(10)算出の手間が大きく、MCNPはライセンスが非現実的

## フェーズ計画

### Phase 0 — 環境（唯一の人間依存ステップ）

- [ ] PHITSライセンス申請・取得（JAEA、研究利用無償。**ユーザー本人の作業**）
- [ ] macOSへのインストールとコマンドライン実行確認（`phits < input.inp` 相当）
- [ ] バージョン・同梱データライブラリ（EGS5モード or デフォルト光子輸送、
      断面積ライブラリの出どころ）を記録する

**完了条件**: サンプル入力がCLIから走り、出力ファイルが得られる。

### Phase 1 — 手書き最小ケースで「地雷」を洗い出す

- [ ] 単色鉛筆ビーム（60 keV）＋水スラブのPHITS入力を**手書き**で作成
- [ ] 一次透過率がBeer-Lambert則および `tests/test_transport.py` の
      viveMonte結果と一致することを確認
- [ ] つまずいた入力形式の罠（単位、密度符号、セル定義、正規化の流儀等）を
      `docs/lessons_learned.md` に記録

**完了条件**: 3者（解析解・viveMonte・PHITS）が統計誤差内で一致した
比較表が残っている。

### Phase 2 — 変換器と出力パーサ

- [ ] `vivemonte/phits_export.py`: scene.yaml → PHITS入力
  - box → RPP、sphere → SPH、cylinder → RCC（重なりの後勝ちルールを
    セル定義の優先順位で再現。ここが最大の設計ポイント）
  - 材料 → `[Material]` セクション（組成はviveMonteと同一ソース
    xraylib/NISTから機械生成し、二重管理しない）
  - SpekPyスペクトル → `e-type` エネルギー分布（同一の離散化を渡す）
  - タリー → `[T-Deposit]`（吸収線量）と `[T-Track]`＋線量換算関数
    （H*(10)）。グリッド定義はVoxelGridと同一に揃える
- [ ] `vivemonte/phits_import.py`: PHITS出力（タリーファイル）のパーサ。
      値と統計誤差の両方を取り出す
- [ ] 各変換の単体テスト（生成された入力テキストのスナップショット照合＋
      境界ケース: 重なり、background材料、回転なし前提の確認）

**完了条件**: `python -m vivemonte phits-export examples/chest_room.yaml`
が人手修正なしでPHITSに通る入力を吐く。

### Phase 3 — 相互検証ハーネス

- [ ] CLI `python -m vivemonte crosscheck scene.yaml` を追加:
      viveMonte実行 → PHITS入力生成 → PHITS実行 → 両者のタリーを
      統計誤差込みで比較し、合否と差分マップを報告
- [ ] 許容基準を**事前に**文書化する（例: 一次透過率は2σかつ2%以内、
      ボクセル線量はカーマ近似・組成規格差による系統差を見込んで
      材料ごとに設定。ICRP/ICRU組成差で数%は正当にずれる —
      lessons_learned既載の教訓を反映）
- [ ] 代表シーン3種で実施: (1) 単純スラブ、(2) chest_room.yaml、
      (3) 鉛遮蔽強調シーン（後方散乱・重元素で差が出やすい条件）
- [ ] 結果をREADMEに反映（「教育・研究用」注記の書き換え、比較表の掲載）

**完了条件**: 3シーンの合否表がリポジトリに残り、READMEが更新されている。

### Phase 4 — スキル化と継続運用

- [ ] 手順が安定したら `/phits-check` 的なスキルに固める
      （入力の落とし穴・実行手順・許容基準をスキル本文に蓄積）
- [ ] 以後の物理モデル追加（蛍光X線等）は「crosscheck合格」を
      マージ条件にする運用を README または CLAUDE.md に明記

## 想定される差異要因（不一致調査の初期チェックリスト）

| 要因 | 方向性 | 対処 |
|---|---|---|
| カーマ近似 vs 電子輸送 | viveMonteが界面近傍で過小/過大 | 診断領域では電子飛程<1mm、ボクセルサイズ≫飛程なら無視可。差が出たらボクセル粗化して確認 |
| 蛍光X線の無視 | 重元素近傍でviveMonteが局所過大・遠方過小 | 既知の未実装項目。差が出れば実装優先度の根拠になる |
| 断面積ライブラリ差（EPDL版数等） | 全域で±1%程度 | 許容基準に織り込む |
| 組成規格差（ICRP vs ICRU-44） | 骨等で数% | 材料ごとの許容誤差（既存 `_LOOSE` 方式） |
| スペクトル離散化・線源の照射野一様近似 | 系統差 | 同一離散スペクトルを両者に渡し線源起因の差を排除 |

## リスクと中止条件

- **ライセンス取得が滞る場合**: Phase 0で停止。代替としてEGS5
  （ひょうご科学技術協会経由で入手容易）への切り替えを検討するが、
  H*(10)算出の手間が増えるためPHITS優先
- **PHITSのmacOS対応が不十分な場合**: Linux VM/別マシンでの実行に切り替え、
  crosscheckは「入力生成＋結果パース」のみローカルで行う疎結合設計にする
  （Phase 2の設計はこの分離を最初から前提とする）
- 変換器がviveMonteの表現力を超える要求（回転体、任意多面体等）を
  受けても**拡張しない**。scene.yamlスキーマの拡張が先

## やらないこと（スコープ外）

- PHITSの中性子・電子線源、RI線源など光子X線管以外の対応
- PHITS入力の全機能カバー（汎用PHITSオペレーターAI化）
- viveMonte側の物理追加（蛍光X線）— 本計画の**後**に、検証網の上で行う
