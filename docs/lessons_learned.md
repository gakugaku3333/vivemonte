# 教訓

## xraylib の `CS_Energy`（μen/ρ）を吸収線量計算に使ってはいけない

`xraylib.CS_Energy` / `CS_Energy_CP` が返す質量エネルギー吸収係数 μen/ρ は、
NIST XAAMDI（Hubbell & Seltzer）の公表値と最大約17%（Al, 60 keVで実測）乖離する。
一方 μ/ρ（全断面積、`CS_Total`/`CS_Total_CP`）はNISTと1%以内で一致する。

**対応**: μ/ρ は xraylib、μen/ρ は NIST XAAMDI を直接取得したCSV
（`scripts/fetch_nist_xaamdi.py` → `chatcarlo/data/nist_xaamdi/`）から
log-log補間で得る2系統構成にした（[materials.py](../chatcarlo/materials.py)）。
`tests/test_materials.py` でNIST公表値とのスポット照合を自動テスト化済み。

**なぜ気づけたか**: 実装直後にNIST公表値と機械的に突き合わせるテストを書いたため。
新しい断面積・線量係数データソースを追加するときは、実装を信用する前に
必ず1次資料の数値と照合するテストを先に書くこと。

## 生体組織の組成規格の違いによるμ/ρの数%ずれは実在する物理差

xraylib の骨データは ICRP 組成、NIST XAAMDI は ICRU-44 組成で、
同じ「骨」でも μ/ρ が最大約2%異なる（60 keVで実測）。
これはバグではなく組成定義の違いによる正当な差なので、
テストの許容誤差は物質ごとに変える（`tests/test_materials.py` の `_LOOSE` 集合）。

## 線量係数（H*(10)等）は記憶で書かず1次資料を取得する

ICRP Publication 74 / ICRU Report 57 の h*(10)/Φ 表は数値を暗記していても
誤る危険が高い防護量データで、公式PDFの数値表は検索だけでは本文抽出できない
ことが多い（WebFetchがPDFのバイナリ内容を正しくテキスト化できないケースが
あった）。OpenMCプロジェクト（MITライセンス）がICRP74値をそのままテキスト
ファイルとして同梱していたため、`gh api repos/openmc-dev/openmc/contents/...`
で直接取得できた（[scripts/fetch_h_star_10.py](../scripts/fetch_h_star_10.py)）。

**対応**: 新しい線量換算係数を追加するときは、(1)記憶から値を書かない、
(2)1次資料PDFが読めない場合はオープンソースプロジェクトが同梱する
転記データ（ライセンス・出典明記の上で）を探す、(3)取得元をコード内
コメントとfetchスクリプトに明記する、の順で対応する。

## 「重元素ほど前方散乱に偏る」という直感は誤り（レイリー散乱の原子形状因子）

原子形状因子 F(Z,q) は q=0 で F=Z、単調減少するが、その減衰の速さは
**軽元素の方が急峻**（xraylibで実測: q=0.5 Å⁻¹でのF/ZはC:0.28, Ca:0.41, Pb:0.59）。
そのため同じ光子エネルギーでは軽元素の方が前方散乱に強く偏り、
重元素は後方まで広がる — 「重い原子ほど前方に強く散乱するはず」という
直感は逆だった。テストを先に直感で書いて失敗し、xraylibの生データを
確認してから修正した（[tests/test_rayleigh.py](../tests/test_rayleigh.py)）。

**対応**: 物理的な直感でテストのアサーションを書く前に、実装が使う
1次データ（この場合はxraylib自体）で数値を直接確認してから書く。
直感と実装のどちらが誤っているか区別がつくまでテストの向きを決め打ちしない。

## mAs絶対値校正でn_historiesを二重に割るバグ（符号・有限性しか見ないテストは桁バグを検出できない）

`chatcarlo run --dose-grid` のmAs校正済み絶対値出力が、実際の期待値より
約`n_histories`倍（実測で約100万倍）小さい値になっていた
（[__main__.py](../chatcarlo/__main__.py)）。原因は
`dose_per_history = dose / n_histories` で既に「1光子あたり」に変換した値に、
さらに `scale = n_photons_real / n_histories` という`n_histories`で割った
係数を掛けていたため。正しくは `scale = n_photons_real` のみでよい
（1光子あたりの値に実光子数を掛ければ絶対値になる）。

発見の経緯: ユーザーから「胸部X線撮影の皮膚線量を計算して」と実際に使ってみた際、
SpekPyの`get_kerma()`（自由空間カーマの参照値）およびChatCarlo自前の
`mu_en_rho`データを使った独立の手計算と突き合わせたところ、桁が大きくずれて
いることに気づいた。既存の`tests/test_calibration.py`はこの絶対値校正の結果に
対して「正の値か・有限か」しかアサートしておらず、桁の間違いを検出できていなかった。

**対応**: 絶対値・物理量を扱うテストは、符号・有限性だけでなく
**独立した参照計算との桁の一致**（今回はSpekPyの`get_kerma()`、factor 50以内）
まで検証する。「実際にツールを使ってみる」というユーザー視点の検証が、
単体テストでは拾えないスケーリングバグの発見につながった
（`tests/test_calibration.py::test_calibrated_dose_matches_spekpy_kerma_order_of_magnitude`）。

## グリッド全体の「最大値」統計は空気ボクセルという非物理的な位置に落ちやすい

`chatcarlo run --dose-grid`が出力する「最大吸収線量」「最大H*(10)」は、
実際に調べると多くの場合**背景（空気）ボクセル**に落ちる。原因は2つあり、
どちらもtrack-length estimatorそのものは正しく、統計として意味のある
「グリッド全体の最大」という統計量の選び方が悪いだけだった:

1. **H*(10)の点線源近傍発散**: 点線源モデルではフルエンスが1/r²で発散する。
   線源直近（chest_room.yamlの実測で線源から3〜19cm）の空気ボクセルが
   最大H*(10)を占め、実測で約1e11 pSv（=100 Sv超）という非物理的な値になった。
   実際のX線管にはhousing・コリメータがあり、この位置に人や検出器が
   存在することはない。
2. **吸収線量の材料境界での後方散乱発散**: 水スラブに鉛筆ビームを垂直入射
   させる単純なシーンでも、最大吸収線量は水の内部ではなく**表面直前の空気
   ボクセル**に落ちた。しかもこれは粗い解像度の離散化誤差ではなく、
   **解像度を細かくするほど値が増大する**現象だった
   （2cm: 1.4e-13, 1cm: 1.5e-13, 0.5cm: 6.4e-13, 0.2cm: 3.5e-12 Gy/history、
   おおよそ1/解像度²で成長 — カーマは密度で正規化済みなので密度の
   アーティファクトではなく、材料境界での後方散乱によるkerma track-length
   タリーの境界効果と考えられる）。「粗いグリッドのせいだろう」という
   直感は誤りで、解像度を振って確認するまで気づけなかった。

**発見の経緯**: 前セッションで「H*(10)最大値が線源近傍で異常に大きい」と
気づいたことをメモリに残しておき、次セッションで検証を依頼された。
実際に最大値ボクセルの位置・材料・線源からの距離を出力するスクリプトを書いて
確認したところ、上記2つの異なる原因が判明した。特に(2)は「解像度を細かくすれば
直る」という直感で終わらせず、実際に4段階の解像度で数値を比較したことで
「離散化誤差ではなく真の発散に近い現象」だと分かった。

**対応**: タリー自体（track-length estimator）は修正していない（物理的に
正しい推定量のため）。代わりに`chatcarlo run`の「最大」統計に診断を追加した:
- `Geometry.nearest_object_distance_cm`: 線源等の点からシーン内の各物体の
  境界ボックスまでの最短距離
- `background_medium_warning`（[diagnostics.py](../chatcarlo/diagnostics.py)）:
  最大吸収線量ボクセルが背景（空気）なら常に警告する（吸収線量は媒質固有の
  量なので、空気中の値は患者・検出器等の実体がある位置の被ばく評価には使えない）
- `near_source_air_warning`: 最大H*(10)ボクセルが背景かつ、シーン内の
  どの物体よりも線源に近い位置にある場合のみ警告する（H*(10)自体は
  面積線量モニタリング用に空気中で評価する量なので、単に「空気だから」
  では警告しない。「物体より線源に近い」という条件は、実在し得ない
  近傍であることの明確な根拠になる）

数値そのものは変更せず（勝手な除外・補正はしない）、位置・材料の診断情報と
警告メッセージだけを追加した。グリッド全体の「最大」を安全に報告できる
統計量として使いたい場合は、この2つの警告が出ないことを確認するか、
評価したい位置（患者表面等）に直接細かいグリッドを敷く方法を使うこと
（`tests/test_dose_diagnostics.py`）。

## チャット添付のインライン表示（render）は自己完結HTMLの`<script>`を実行しないことがある

`chatcarlo trace`/`preview`が生成する自己完結HTML（vanilla JS + canvas）を
チャットの添付ツールで`display: "render"`（インライン表示）で送ったところ、
ユーザー側で「何も表示されない」という報告があった。実際にブラウザ
（ローカルHTTPサーバ経由でこのツールのBrowser機能から開く）で検証すると
コンソールエラーなく正しく描画され、コード側の不具合ではなかった。

**原因**: チャットのインラインプレビュー枠は、埋め込みJSを実行しない/制限
する場合がある（サンドボックスの`<script>`実行制約と推測、明確な仕様は
不明）。ファイルの中身自体は正しくても、この経路では空白キャンバスになる。

**対応**: ChatCarloが生成するインタラクティブHTML（trace/preview）は
`display: "attach"`（ダウンロードカード）で送り、実ブラウザで開いてもらう
運用にした。`.claude/skills/vive-check/SKILL.md`の「`open <html>`でブラウザ
表示」という既存の指示もこの前提に基づいている——チャット添付プレビューは
静的な検証手段としては信用しないこと。真に描画を検証したいときは、
このツールのBrowser機能（`preview_start`+`navigate`、file://は使えないため
`python -m http.server`でローカル配信してから開く）でコンソールエラーの
有無まで確認する。

## 新しい物理モデル（CTDIvol校正・ヒール効果）もSpekPyの実挙動を先に実測してから実装する

CTDIvol校正・ヒール効果の実装前に、SpekPyの軸外計算（`Spek(x=...)`）の
座標系（どちらが陽極側か）を記憶で仮定せず、`x`を振ってフルエンス・平均
エネルギーを実測してから設計した（x<0が陽極側であることを確認）。
既存の教訓（xraylib CS_Energy、h*(10)係数、レイリー原子形状因子）と同じ
パターンで、外部ライブラリの規約は使う前に必ず実測すること。

**副産物のバグ**: ヒール効果のエネルギー抽選で、陽極カットオフ境界を
補間する際に受理された光子がゼロ放出ビンに落ちて`rng.choice`の確率が
合計1にならず例外になるケースがあった（`tests/test_heel_effect.py::
test_heel_composes_with_cone_and_rotation`で発覚）。**対応**: ビンindexを
非ゼロフルエンスの範囲にクランプ（`chatcarlo/source.py`）。境界条件・
カットオフのような「稀にしか踏まないパス」は、通常の物理的検証テストとは
別に、境界そのものを狙ったテストケースを書かないと見つからない。

## `set()`の反復順に依存すると、同一seedでも実行のたびに結果が変わる

`transport.py`を`transport.py`/`source.py`/`spectrum.py`/`physics.py`/
`trajectory.py`/`diagnostics.py`に分割するリファクタリング（2026-07-14）の
検証中、**リファクタリング前から** `chatcarlo run --seed 42` の出力が
実行のたびに微妙に変わるバグが既存していたことが判明した。

原因: 材料名でバッチをグループ化する箇所（`_linear_mu_batch`等、5箇所）が
すべて`for name in set(materials.tolist())`で反復しており、Pythonの文字列
ハッシュはプロセスごとにランダム化される（`PYTHONHASHSEED`）ため、
グループを処理する順序がプロセスごとに変わっていた。レイリー元素抽選など
グループ処理の順に依存して乱数を消費する経路があるため、抽選結果自体が
ずれて`--seed`を固定しても再現しない。

**対応**: グループ化を`materials.material_groups()`に一本化し、内部で
`sorted(set(...))`にソートを固定した。異なる`PYTHONHASHSEED`の2プロセスで
同一seedの結果が一致することを`tests/test_reproducibility.py`で回帰
テスト化した。

**教訓**: `set`や辞書のキー反復順は「その場では動くが再現性がない」典型
パターン。乱数を消費するループでバッチを名前・IDでグループ化するときは、
反復順を明示的に固定（ソート等）しないと、シード固定による再現性の
保証が静かに壊れる。この手のバグはテストが通っていても気づけない
（テストはseedを毎回変えないため、たまたま同じ順序が出続けて隠れる）。
再現性が要件のコードでは「同一seedを2回実行して完全一致するか」を
定期的に手で確認する価値がある。

## コード間の数値差は、まず「物理モデルが揃っているか」を疑う（EGS5相互検証, 2026-07-15）

PHITSライセンス取得が抄録締切（2026-07-17）に間に合わないため、KEK配布の
EGS5（登録不要で即日入手可）に切り替えて相互検証を行った際、初回実行で
ChatCarloとEGS5の一次透過率が相対4.7%（約8.6σ）ずれ、独立監査
（vive-auditor）で差し戻しになった。

**原因**: EGS5チュートリアル（`tutor5`）のデフォルトPEGS5設定が
`IBOUND=0`（自由電子Klein-Nishina全断面積）だったのに対し、ChatCarlo/xraylib
の`CS_Compt`はEPDL由来の**束縛電子補正込み**断面積を使っていた。60 keV水では
コンプトンが全断面積の86%を占めるため、この物理モデル差だけで透過率が
数%動く。「断面積ライブラリの版数差（±1%程度）」という説明は、支配的な
相互作用チャネルの寄与を考えれば数値的に足りないと気づけたはずだった。

**対応**: 自由電子KN全断面積を解析式から独立計算し、xraylibのEPDL値
（束縛補正込み）と比較して+3.03%の乖離を確認。EGS5側を`IBOUND=1`
（束縛コンプトン）に揃えて再実行したところ、相対0.88%・約1.7σまで縮み、
計画書の事前基準（2σかつ2%以内）に適合した（詳細:
[egs5_crosscheck/RESULTS.md](egs5_crosscheck/RESULTS.md)）。

**教訓**: 独立実装のコード同士を比較検証するときの不一致は、まず
「支配的な相互作用チャネルについて、両者が同じ物理モデル（自由電子 vs
束縛電子、コヒーレント散乱の有無等）を使っているか」を最初にチェックする。
「ライブラリの版数差」のような曖昧な説明で±数%の系統差を片付ける前に、
その相互作用が全断面積に占める割合から「その説明で数値の桁が合うか」を
概算すること。コード間比較（物理モデルを揃えて実施）は輸送アルゴリズム・
実装の検証であり、断面積データそのものの正しさは別途一次資料
（NIST等）との照合で担保する、という役割分担を混同しないこと。

## legacy Fortran（EGS5等）をモダンgfortranでビルドするには明示的な緩和フラグが要る

Homebrew gfortran 16.1.0はデフォルトで、F77の暗黙インターフェース
（関数と宣言の食い違い、例: `cprfil`/`alin`/`alini`/`pairfr`/`pairfz`/
`brmsfz`/`bremfr`/`bremfz`/`QFIT`）をエラーにし、古いgfortran/g77では
通っていたコードのビルドが失敗する。

**対応**: EGS5の`egs5run`スクリプトの`gfort`マシンプロファイルの`CFLAGS`に
`-std=legacy -fallow-argument-mismatch -w`を追加（デフォルトは
`-fno-automatic`のみ）。これでtutor5・水スラブ検証コードとも問題なく
ビルド・実行できた。

**教訓**: 配布年代の古いFortran資産を新しい環境でビルドするときは、まず
コンパイラのデフォルト厳格化（インターフェースチェック強化等）を疑い、
`-std=legacy`系のフラグを最初から試す。エラーメッセージの個々の関数名を
一つずつ潰そうとするより早い。

## 全断面積が一致しても、エネルギー付与（kerma）の絶対値まで一致するとは限らない（EGS5相互検証Phase 2b, 2026-07-16）

Phase 1（一次透過率、相対0.88%一致）・Phase 2a（BSF、EGS5側の測定不確実性で保留）に
続けて、水ファントムのPDD＋側方プロファイル（47ビン、per-history絶対値Gy/history）を
ChatCarlo・EGS5で相互検証したところ、**EGS5が全47ビンで一貫してChatCarloより
約−1.71%低い**という、統計揺らぎでは説明できない系統差が見つかった
（詳細: [egs5_crosscheck/PDD_RESULTS.md](egs5_crosscheck/PDD_RESULTS.md)）。
vive-auditorの独立監査で数値・判定は完全に再現され（条件付き合格）、原因は
未確定のまま次の課題として持ち越した。

**分かったこと**:
- **形状（深さ依存・側方依存の相対的なプロファイル）は両コードで完全に一致**する。
  各コードを自分自身の基準ボクセルで100%正規化して形状だけを取り出すと、
  47ビン中2σを超える有意差は実質ゼロ（多重比較を考慮すればノイズの範囲）。
  問題は**絶対値スケールだけ**に局在している。
- **有力仮説（未検証）**: ChatCarloのkerma推定量は各飛行区間でNIST XAAMDI
  （Hubbell-Seltzer）のμen/ρを直接テーブル参照する「係数ベース」。EGS5の
  ausgabは、PEGS5が生成した断面積データ（Storm-Israel系＋IBOUND束縛コンプトン
  補正）に基づく実際のモンテカルロ相互作用から積み上げた「シミュレーション
  ベース」の局所エネルギー付与で、**NIST XAAMDIとは独立に評価された別データ
  ソース**である。Phase 1で検証済みの「全断面積・透過率が0.88%で一致」は
  低Z水で支配的なコンプトン散乱確率で決まる量であり、光電吸収の断面積に
  数%規模のずれがあっても透過率にはほとんど影響しない。しかし光電吸収は
  相互作用1回あたりほぼ全エネルギーを局所付与するため、**エネルギー付与
  （kerma/dose）の比較では、透過率の比較より断面積の食い違いが増幅されて
  見える**可能性がある。
- ChatCarlo側のμen/ρ参照値そのもの（NIST XAAMDIテーブル読み取り）にデータ
  入力ミスはないことをファイル・単位換算・σ計算まで独立に検算して確認済み。

**教訓**: 「コード間で全断面積・一次透過率が良く一致した」ことは、「エネルギー
付与（kerma/dose）の絶対値も同精度で一致する」ことを保証しない。両者は
異なる物理量（総相互作用確率 vs 相互作用種別ごとのエネルギー移行の内訳）で
あり、後者は前者よりデータソース間の食い違いに敏感になりうる。線量計算コードの
相互検証では、透過率だけでなくエネルギー付与量そのものを独立に照合する
フェーズを必ず設けること（Phase 1だけで「検証完了」と判断しない）。

**続報（2026-07-17、原因究明完了）**: 上記の「有力仮説（未検証）」を
[plan_compton_transfer_check.md](egs5_crosscheck/plan_compton_transfer_check.md)で
検証したところ、**光電吸収断面積の食い違いという当初の仮説は棄却**された
（PEGS5内部データ vs xraylib/EPDLの光電断面積差は+0.408%で、kerma全体への
影響は+0.19%と観測方向と符号が逆。「部分的要因」の条件すら満たさない）。
代わりに**コンプトン散乱のエネルギー移行分布の差**が部分的要因（観測−1.71%の
約64%を説明）と判明した。EGS5側のPEGS5設定`IBOUND=1, INCOH=0`は「全断面積の
みStorm-Israel系の束縛補正を適用し、散乱光子のエネルギー・角度抽選は自由電子
Klein-Nishinaのまま（S(q)補正の棄却法なし）」という**混成構成**であり、
ChatCarlo/NIST XAAMDIはS(q)補正込みの散乱理論で評価されているため、平均
エネルギー移行率に系統差（60 keVで−2.35%）が生じる。机上検算（自由電子KNの
解析的平均移行率）・EGS5ソースコード裏取り（`egs5_compt.f`の棄却関数が
実際にS(q)補正なしで動くことをコード行で確認）・EGS5計装再実行（実際の
コンプトンイベント900万件超から⟨T⟩を直接測定し机上予測と0.48σで一致）の
三重の独立手法で確認済み（vive-auditor監査も条件付き合格）。残り約36%は
依然未解明。

**教訓（追加）**: 「断面積（相互作用が起こる確率）が高精度で一致している」
ことは、「その相互作用でどれだけのエネルギーが移行するかの分布まで一致する」
ことを含意しない。EGS5のIBOUND（全断面積の束縛補正）とINCOH（散乱光子の
エネルギー・角度抽選への束縛補正）は独立なフラグで、片方だけ有効にする
「断面積のみ束縛・運動学は自由電子」という混成構成は現実的なユースケース
だが、kerma/線量レベルの精密比較では数%規模の見かけの系統差を生みうる。
コード間クロスチェックで物理モデルを揃える際は、「断面積を揃えたから
物理モデルは同じ」と即断せず、断面積と運動学（角度・エネルギー分布）を
別々のチェック項目として確認すること。

**続報2（2026-07-17、残差調査完了）**: 残り約36%の未解明分を
[plan_residual_check.md](egs5_crosscheck/plan_residual_check.md)で調査したところ、
2つの独立要因の合算でほぼ説明できることが判明した。

1. 単一エネルギー(60 keV一次成分のみ)の机上検算では見えなかったが、実際の
   散乱スペクトル（多重散乱で軟化した二次光子を含む）まで畳み込むと、
   「コンプトン移行分布差」単独の寄与は−0.75%程度（60 keV一次成分のみの
   −1.095%より小さい）にとどまった。
2. 代わりに、**ChatCarlo自身のμen/ρ log-log補間**に、想定していなかった
   系統誤差が見つかった。NIST XAAMDIの公表データは30-40-50-60-80 keVという
   疎な格子しか持たず（この帯域でμen/ρ(E)は極小を持つ凸型の曲線）、格子点
   そのものでは補間なしの第一原理近似と0.02-0.08%で一致するにもかかわらず、
   格子点の中間（35/45/55 keV）ではlog-log線形補間が実際の値より+1.5〜1.8%
   高く出ることを直接確認した。
3. EGS5をINCOH=1（S(q)束縛補正込みのコンプトン散乱、ChatCarloの物理と
   整合する構成）で再実行したところ、47ビン平均相対差は−1.71%から−0.633%まで
   縮小し、事前登録した予測値（コンプトン移行分布差の解消＋補間アーティファクト
   の残存を織り込んだ−0.81%±0.3pt）の範囲内に収まった。密度・断面積の
   微小差（候補として検討）は最深部でも0.19%pt程度で無視できる規模だった。

**教訓（追加2）**: ①単一エネルギー・単一相互作用種別だけの机上検算は、
多重散乱で生じる二次光子スペクトル全体を通した効果を過小・過大どちらにも
見誤りうる。可能なら実際のスペクトルで重み付けした畳み込み計算まで行う
（ChatCarlo自身の輸送結果を重みに使う近似でも一定の説明力がある）。
②離散テーブルを補間して使う物理量（今回はμen/ρ）は、格子点そのものでの
検証だけでは不十分——格子点の**中間**で検証しないと、曲率由来の補間誤差
（今回は水で+1.5〜1.8%、無視できない規模）を見逃す。これはEGS5比較固有ではなく、
水の場合ChatCarlo全体の線量計算精度に関わる限界であり、30〜80 keV帯を含む
シミュレーションでは常にこの影響を受けうる。他材質も同型の疎な格子・凸曲線を
持つため方向性は共通すると推測されるが、バイアス幅が水と同程度かは未確認
（水以外での定量評価・対処は本調査のスコープ外、今後の課題）。

## 推定量の種類が違うと、同じ物理量でも履歴あたりの統計誤差が桁違いになる（EGS5相互検証Phase 2b, 2026-07-16）

Phase 2bでChatCarlo（n=5,000,000）とEGS5（n=100,000,000、20倍の史数）を比較したところ、
薄い（1cm×2×2cm程度の）スコアビンにおいてChatCarloの統計誤差がEGS5の単純な
1/√nスケーリング予測（史数差から期待される値）よりずっと小さい（履歴あたり
分散でおよそ12〜14倍小さい）ことが分かった。バグではなく、推定量の種類の違い
による既知のMC効果:
- **track-length（飛程長）推定量**（ChatCarlo）: ビンを通過した光子**全員**に
  ついて、飛行距離×μen/ρ×エネルギーを決定論的に積算する。相互作用の有無に
  関係なく通過さえすれば必ず寄与するため、履歴あたりの分散が小さい。
- **collision（衝突点）推定量**（EGS5のausgab）: ビン内で実際に相互作用が
  起きた光子だけが寄与する。薄いビンでは通過光子のうちごく一部しか実際に
  相互作用しないため、「ほとんど0、稀に大きい値」という当たり外れの激しい
  分布になり、履歴あたりの分散が大きい。

**教訓**: 異なるコード・異なるタリー方式で統計誤差を比較するとき、史数比から
単純に期待される誤差比と大きくずれていても、それだけでは異常のサインにならない。
両者が同じタリー技法（track-length vs collision）を使っているかを先に確認する。
逆に、薄い領域での精密なタリーが必要な場面では、可能ならcollision推定量ではなく
track-length推定量を選ぶ方が、同じ史数で大幅に統計精度を稼げる
（`chatcarlo/tally.py`のcollision/track-length二推定量設計、Phase 2aの
BSFスクリプトの選択と同じ理由）。

## track-lengthタリーの「サブステップ中点」スコアリングは、区間始点が境界に
## 位相同期すると系統バイアスを生む（会話ワークフロー実演, 2026-07-16）

学会抄録用に「vive-interview→vive-checkの会話ワークフローだけで、EGS5相互検証
（Phase 2b）と同じ結論に行き着く」実演を作る過程で、新機能`source.field.shape:
parallel`（非発散の平行ビーム）を追加した。この機能で組んだシーン（全光子が
水ファントム前面ちょうどy=0から出発）を`--dose-grid`で計算したところ、
vive-auditorの独立監査（同一乱数トラックに対する厳密box-overlap推定量との
並行計算）で、**ファントム表面ボクセル層だけが約−2.7%系統的に過小評価**
されていることが発覚した（[egs5_crosscheck](egs5_crosscheck/)関連ドキュメント参照）。

**原因**: `accumulate_track_length`（[tally.py](../chatcarlo/tally.py)）は
区間をサブステップに等分し、各サブステップの**中点**（決定的な位置）が属する
ボクセルへスコアしていた。区間の始点がボクセル境界ちょうどに揃わない通常の
シーン（点線源からの発散ビームは空気中を経てから材料境界に達するため、
最初の区間は境界ちょうどからは始まらない）では、サブステップ中点の量子化誤差は
光子ごとにバラバラな位相を持ち、平均すると打ち消し合う。しかし今回の平行ビームは
**全光子の第1区間が同じ座標（材料境界）から始まる**ため、量子化誤差の位相が
全光子で揃い、打ち消されずに系統誤差として残った（表面ボクセル層のみ、
1/解像度で発散しない新種のパターン）。既存の`background_medium_warning`/
`near_source_air_warning`（[diagnostics.py](../chatcarlo/diagnostics.py)）は
どちらのボクセルも「水」で発火しない設計のため、この種のバグは盲点だった。

**対応**: サブステップの中点ではなく、各サブステップ内の**層化乱数点**で
スコアするよう変更した。これで区間の始点位置に関わらず、任意のボクセルへの
期待寄与が厳密な幾何学的重なり長に一致する（不偏推定量）。修正はスコアリング
（空間分配）だけに限定され、区間全体のエネルギー積算（Σweight×length）は
乱数点の位置と無関係に厳密なままなので、輸送本体の結果（吸収/脱出割合、
相互作用サンプリング）には一切影響しない設計にした（`transport_photons`に
`tally_rng`引数を追加、`rng.spawn()`で輸送用乱数列と分離）。

**教訓**:
- 決定論的な離散化（中点・格子点など）を使う推定量は、入力データの位相が
  たまたま離散化格子と同期する条件で系統バイアスを生みうる。「ランダムな
  入力なら中点近似は十分」という直感は、入力の分布が本当にランダムである
  前提が崩れると成り立たない。層化乱数点は原理的にこの種のバイアスを消せる。
- 新しいソースジオメトリ（今回は非発散の平行ビーム）を追加するときは、
  それが既存の暗黙の前提（「光子は必ず空気中から材料に入射する」等）を
  破っていないかを疑うこと。この種の前提破りは、通常のシーンでの回帰テストでは
  絶対に検出できない。
- 独立監査（vive-auditor）が「同一乱数トラックに対する2つの推定量の並行比較」
  という決定論的な検証手法を使ったことが発見の鍵だった。統計誤差に埋もれる
  規模のバイアス（今回は表面層のみ−2.7%）は、単純な数値比較や集計値の
  照合だけでは見逃しやすい。
定期的に手で確認する価値がある。

## 疎な参照テーブルのlog-log「線形」補間は、格子点で一致していても
## 格子点間で無視できない曲率誤差を持ちうる（EGS5相互検証Phase 2b残差調査, 2026-07-17）

Phase 2b（PDD/OCR相互検証）で観測された−1.71%系統差の原因究明の過程で、
`chatcarlo/materials.py`の`mu_en_rho()`（NIST XAAMDIテーブルのμen/ρをlog-log
線形補間で参照）に、格子点そのものでは0.1%未満で一致するにもかかわらず、
格子点間の中間エネルギーでは最大3.3%（60〜80keVの20keV間隔区間で最悪）の
曲率誤差があることが判明した。原因は「格子点で正しい＝補間全体で正しい」という
暗黙の前提の誤り。XAAMDIテーブルは対数対数でもなお緩やかに曲がっており、
特に格子間隔が広い（20keV前後）区間でlog-log線形補間の直線近似が実際のカーブから
乖離する。

**検出方法**: 独立に評価できる「第一原理近似」（xraylib photo+compton×
Klein-Nishina/S(q)平均エネルギー移行率、格子点で0.02〜0.08%の高精度一致を
別途確認済み）を基準に、格子点間の中間エネルギーでテーブル補間値と比較した。
単純に「補間値は両隣の値の間にある」というテスト（`test_loglog_interpolation_
between_grid_points`、既存）だけでは検出できない種類のバグで、外部の独立な
参照値との比較が必要だった。

**対応**: `mu_en_rho()`をlog-log線形補間からlog-log PCHIP（区分3次エルミート、
形状保存・格子点は厳密通過）に変更した。30〜80keV帯の最大曲率誤差は3.3%→0.6%
（82%削減）。実際のPDD/OCRスペクトルで畳み込んだ寄与も改善を確認し、Phase 2bの
EGS5相互検証残差が−1.71%→−0.91%まで縮小したことで実効果を検証した
（詳細: [egs5_crosscheck/PDD_RESULTS.md](egs5_crosscheck/PDD_RESULTS.md)）。

**教訓**:
- 疎な参照テーブル（格子間隔が広い、対数軸でもなお曲がっている量）をlog-log
  線形補間で使う設計は、格子点での検証だけでは補間全体の精度を保証しない。
  格子点間の中間値を独立な参照値と比較するテストを別途用意すべき。
- 「格子点一致のテスト」と「補間区間全体の精度」は別物として扱う。今回は
  前者（`test_mu_en_rho_matches_nist`）は既にパスしていたが、後者の欠陥が
  1年近く未検出のまま残っていた。
- 形状保存補間（PCHIP等）は、格子点を厳密に通過しつつ区間内の曲率を線形補間
  より正確に再現できるため、線形補間からの置き換えとして低リスク・高効果な
  選択肢になりやすい（外挿はしない・オーバーシュートしない特性を持つ）。
- この種のバグは、コード単体のテストでは検出しにくく、**独立コード
  （EGS5）との定量的なクロスチェックが実効的な検出手段になった**。
  Phase 1・2aのような「同一コード内の比・全断面積の粗い一致確認」だけでは
  見えなかった精度課題が、Phase 2bの「絶対値同士の直接比較」で初めて
  露見した（[pdd60_NOTES.md](egs5_crosscheck/pdd60_NOTES.md)の「未確認事項」
  節も参照）。
