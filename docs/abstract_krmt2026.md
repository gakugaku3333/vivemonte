## 現在のドラフト（392字/400字, 2026-07-16更新 — 「会話」「話しかける」等の表現を音声と誤解されない「チャット」に統一、確定版）

**演題名**: 対話型AIで行えるモンテカルロ計算システムの開発

```
【目的】モンテカルロ法は放射線分野で有用だが、専門的なプログラミング知識が必要で参入障壁が高い。誰もがチャットだけでシミュレーションを行える環境の実現を目指し、システムを開発した。
【方法】Claude Codeを利用し、Pythonで診断X線領域の光子輸送モンテカルロコードと、それをチャットで操作するワークフローを構築し、確立コードEGS5との比較で計算精度を評価した。
【結果】「水ファントムに60keVの光子の線量分布を計算したい」とチャットで伝えると、AIが幾何学条件を確認し、線源と水ファントムを配置し、3D画像で確認後に計算を実行、線量分布を提示する一連の流れを実現した。EGS5との比較はPDD平均+0.10%・最大0.44%、OCR平均+0.15%・最大1.25%で一致し、物理的妥当性を確認した。
【結論】プログラミング知識がなくてもAIとのチャットでモンテカルロ計算を実行できるシステムを構築できた。
```

（「話しかける」「対話」「会話」は音声インタラクションと混同されうるため「チャット」に統一。
「物理的妥当性」の表現は、EGS5という実測検証済みの確立コードとの相互検証を根拠にした慣用的な表現であり
問題ないと判断し現状維持——検討の経緯は本ファイル末尾を参照。
%値はEGS5・ChatCarloそれぞれ自身の基準ボクセルで正規化した形状の差分。詳細は
[egs5_crosscheck/CONVERSATIONAL_WORKFLOW_DEMO.md](egs5_crosscheck/CONVERSATIONAL_WORKFLOW_DEMO.md)・
[egs5_crosscheck/PDD_RESULTS.md](egs5_crosscheck/PDD_RESULTS.md)を参照。なお正規化前の絶対値スケールには
未解明の約1.7%の系統差があり、抄録では字数制約上「形状の一致」のみを記載——詳細は次項）

## 経緯（方向性の絞り込み）

ユーザーの初期アイデア（「モンテカルロ法の民主化」「対話でシミュレーションコードを作れる」）
を軸に、対話しながら5案を検討した:

1. 民主化を前面に出す（参入障壁の除去）
2. 開発速度の実証（3日・約2,600行+テスト1,500行という定量事実で語る）
3. 検証方法論を主役にする（教訓集にある「AIの記憶由来の係数が最大17%ずれた」
   「桁バグが混入したが検証で捕まえた」という話を中心に据える）
4. 運用ワークフロー（コード開発ではなく、会話でシミュレーションを"使う"体験）
5. 民主化+検証の折衷

**採用: 案1（目的・方法の骨格）＋ 案4（結果の具体例）の統合**。理由はユーザー指定
（「この方法で以下のものを作った」という組み合わせ方の指示）。

### 途中で修正した構造上の問題

初期ドラフトの【結果】は「一次線透過率が解析解と一致し、物理的妥当性を確認した」と
書いていたが、これはユーザーから
**「コード開発とシミュレーション行為が混ざっている」**と指摘された。

この抄録が検証しているのは「対話だけでコード開発できるか」（開発プロセスの問い）で
あって、「このツールの計算精度が正しいか」（物理的妥当性の問い、PHITS比較の領分）
ではない。両者を混同すると、読み手にとって何を主張している研究なのかが曖昧になる。

**対応**: 【結果】に含む物理関連の記述は、常に「対話だけでここまで構築できた」という
**開発プロセスの成果**として書き、「シミュレーション結果が正しい」という**独立した
物理的主張**にはしない。今回の最終稿では、Beer-Lambert/NIST照合のくだり自体を削り、
案4の会話ワークフロー実例（ヒアリング→配置→3D確認→計算→提示）に差し替えることで
この混同を解消した。

## 今後の作業（次セッション）

1. **相互検証は実施済み・【結論】反映済み（2026-07-16、比較対象をPhase 2b PDD/OCRに変更）**。
   PHITSは国内でもライセンス申請から配布まで約3週間かかり締切に間に合わないため、
   即日入手可能な確立コードEGS5（KEK配布）との比較に切り替えた。
   - **Phase 1（一次透過率、当初の結論文で採用）**: 60 keV鉛筆ビーム＋水スラブ10cm、
     束縛コンプトン断面積（IBOUND=1）を揃えた条件でChatCarlo 12.589% vs
     EGS5 12.70%（相対0.88%、約1.7σ）、事前基準「2σかつ2%以内」に適合。
     詳細は[egs5_crosscheck/RESULTS.md](egs5_crosscheck/RESULTS.md)。
   - **Phase 2b（水ファントムPDD・側方プロファイルOCR、現在の結論文で採用）**:
     各コード自身の基準ボクセルで正規化した形状比較で、PDD 15ビン中0本・
     OCR 10cm深16ビン中0本・OCR表面近傍16ビン中2本（多重比較の偶然と判断）が
     2σを超え、形状は両コード一致と結論。ただし正規化前の絶対値スケールには
     EGS5が全47ビンで一貫して低い−1.71%の系統差があり、原因（cross-section
     データソース差が有力仮説）は未解明のまま。抄録は字数制約から「形状一致・
     物理的妥当性確認」のみを記載し、絶対値スケールの未解明差には触れない
     （虚偽ではないが、粒度を落とした記載である点に注意）。詳細は
     [egs5_crosscheck/PDD_RESULTS.md](egs5_crosscheck/PDD_RESULTS.md)。
   - Phase 1は初回（自由電子KN, IBOUND=0）が相対4.7%の差で独立監査（vive-auditor）
     が差し戻し、真因が断面積の物理モデル差（束縛補正の有無）と判明したため
     再実行して基準適合に至った経緯あり。Phase 2bはvive-auditorが数値・判定を
     再現の上「条件付き合格」（原因分析は未検証と明記する条件）。
   - **【結果】【結論】の内容は、2026-07-16に実際にvive-interview→vive-check
     ワークフローで再現実演済み**（`examples/water_phantom_pdd_ocr.yaml`、
     [egs5_crosscheck/CONVERSATIONAL_WORKFLOW_DEMO.md](egs5_crosscheck/CONVERSATIONAL_WORKFLOW_DEMO.md)）。
     単色ビーム(`source.spectrum`)・平行ビーム(`field.shape: parallel`)を
     chatcarlo本体の新機能として実装し、会話ワークフローで組んだ標準CLI計算が
     PDD/OCR形状でEGS5と正規化差0.1〜0.2%pt程度で一致することを独立に確認した
     （実演中にvive-auditorのステージB監査で`--dose-grid`のtrack-lengthタリーの
     実在バグ——境界始点の位相同期による表面層−2.7%系統誤差——を発見、修正、
     再監査で条件付き合格。教訓は[lessons_learned.md](../lessons_learned.md)）。
   PHITS比較は11月の本発表までの追加データとし、ライセンス申請は別途進める。
