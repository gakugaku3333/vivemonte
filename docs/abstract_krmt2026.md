## 現在のドラフト（376字/400字, 2026-07-15時点）

**演題名**: 対話型AIで行えるモンテカルロ計算システムの開発

```
【目的】モンテカルロ法は放射線分野で有用だが、専門的なプログラミング知識が必要で参入障壁が高い。誰もが会話だけでシミュレーションを行える環境の実現を目指し、システムを開発した。
【方法】開発者はソースコードを記述せず、対話型生成AI(Claude Code)との自然言語の対話のみで、診断X線領域の光子輸送モンテカルロコードと、それを会話で操作するワークフローを構築した。
【結果】「胸部撮影室の散乱線分布を知りたい」と話しかけると、AIが撮影条件を聞き取り、体幹ファントムと撮影室を配置し、3D画像で配置を確認後に計算を実行、線量分布を提示する一連の流れを会話のみで行えた。
【結論】対話型AIにより、プログラミング知識がなくてもモンテカルロ計算システムを自作し会話で運用できることが示された。今後、計算精度についてPHITSとの比較検証を行う予定である。
```

## 経緯（方向性の絞り込み）

ユーザーの初期アイデア（「モンテカルロ法の民主化」「対話でシミュレーションコードを作れる」）
を軸に、対話しながら5案を検討した:

1. 民主化を前面に出す（参入障壁の除去）
2. 開発速度の実証（3日・約2,600行+テスト1,500行という定量事実で語る）
3. 検証方法論を主役にする（教訓集にある「AIの記憶由来の係数が最大17%ずれた」
   「桁バグが混入したが検証で捕まえた」という話を中心に据える）
4. 運用ワークフロー（コード開発ではなく、会話でシミュレーションを"使う"体験）
5. 民主化+検証の折衷

**採用: 案1（目的・方法の骨格）＋ 案4（結果の具体例）の統合**。理由はユーザー指定
（「この方法で以下のものを作った」という組み合わせ方の指示）。

### 途中で修正した構造上の問題

初期ドラフトの【結果】は「一次線透過率が解析解と一致し、物理的妥当性を確認した」と
書いていたが、これはユーザーから
**「コード開発とシミュレーション行為が混ざっている」**と指摘された。

この抄録が検証しているのは「対話だけでコード開発できるか」（開発プロセスの問い）で
あって、「このツールの計算精度が正しいか」（物理的妥当性の問い、PHITS比較の領分）
ではない。両者を混同すると、読み手にとって何を主張している研究なのかが曖昧になる。

**対応**: 【結果】に含む物理関連の記述は、常に「対話だけでここまで構築できた」という
**開発プロセスの成果**として書き、「シミュレーション結果が正しい」という**独立した
物理的主張**にはしない。今回の最終稿では、Beer-Lambert/NIST照合のくだり自体を削り、
案4の会話ワークフロー実例（ヒアリング→配置→3D確認→計算→提示）に差し替えることで
この混同を解消した。
