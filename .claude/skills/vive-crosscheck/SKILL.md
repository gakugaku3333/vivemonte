# vive-crosscheck — 独立コードとの相互検証

引数: 検証したい量（例:「60 keV水スラブの一次透過率」「胸部BSF」「水ファントムPDD」）。
対応するscene.yamlがあれば併せて渡す。

このスキルは**進行と記録の型**だけを担う。EGS5側の実装作業
（ユーザーコード作成・ビルド・実行）は`egs5-operator`エージェントに委任し、
結果の監査は`vive-audit`スキル経由で`vive-auditor`に委任する。メイン会話では
ChatCarlo側の実行と3者比較の取りまとめ、記録の作成だけを行う。

確立コードはEGS5（即日入手可・登録不要）。PHITS導入は2026-07-17にユーザーの
判断で取りやめ、EGS5を継続使用する（経緯: `docs/plan_egs5_crosscheck.md`）。

## 手順

### 1. 事前登録: 許容基準を実行前に固定する

- 比較する量・条件（幾何、線質、統計目標）を明文化する
- 合否基準を**結果を見る前に**決めて書き留める（前例: 一次透過率は2σかつ2%以内、
  ビン単位の線量分布は統計誤差1%未満のビンに限り2σかつ相対2%以内）
- 「一度に1つの差だけを見る」原則を守る（Phase 1のIBOUND教訓）: 物理モデル・
  正規化方式・グリッド定義など複数の変更点を同時に比較条件へ混ぜない
- 既知の系統差要因を先にチェックリスト化する（`docs/lessons_learned.md`と
  下表「想定される差異要因」を必ず読む）

### 2. ChatCarlo側を実行

- 対応するscene.yamlがなければ`vive-interview`または手書きで用意する
- `chatcarlo run`（必要なら`--dose-grid`）で結果を得る。コマンド・seed・
  history数を再現用に記録する
- 境界アーティファクトが疑われる量（浅い深さの線量分布等）は、既存のtrack-length
  タリーの境界バイアスの教訓（`docs/lessons_learned.md`）を踏まえ、必要に応じて
  専用の推定量を使う

### 3. EGS5側をegs5-operatorに委任

- Agentツールで`subagent_type: "egs5-operator"`を起動する
- 依頼文には必ず含める:
  - 検証したい幾何・線質・統計目標・**許容基準**（手順1で固定したもの）
  - **IBOUND=1（束縛コンプトン）固定**の指示（ChatCarloと物理モデルを揃える
    必須事項。揃えないと60 keV水で相対4.7%もずれる — `egs5-operator.md`参照）
  - ChatCarlo側の結果（比較用の数値・統計誤差）
  - ゼロから書かせず`$EGS5/tutorcodes/`の近いチュートリアルを起点にする指示

### 4. 三つ巴比較

- 解析解/文献値・ChatCarlo・EGS5の3列比較表を作る（値±統計誤差）
- 相対差とσ数を計算し、手順1の事前基準と照合する
- 基準に対して不合格でも基準そのものを緩めない。原因調査は別項目として記録し、
  未解決のまま「条件付き合格」で残すことは許容する（前例: Phase 2b、
  `docs/egs5_crosscheck/PDD_RESULTS.md`）

### 5. vive-auditで監査

- `vive-audit`スキルを呼び、ステージB相当（実行結果監査）としてChatCarlo側の
  独立検証を依頼する。EGS5側の再現性確認（同一seed再実行等）も対象に含めてよい
- 差し戻しなら原因調査してから再監査する。監査官の指摘に反証がある場合のみ
  反証根拠を添えて再監査に回す（水掛け論はユーザーに裁定を仰ぐ）

### 6. 記録

- `docs/egs5_crosscheck/<name>/RESULTS.md`に残す（前例:
  `docs/egs5_crosscheck/PDD_RESULTS.md`、`docs/egs5_crosscheck/RESULTS.md`）:
  比較表、相対差・σ数、事前基準との照合結果、既知の残存系統差、未確認事項
- 自作の`.f`/`.inp`/実行ログも同ディレクトリにコピーして追跡する
  （EGS5本体はStanford/SLACライセンスの第三者コードにつきgitignore対象、
  差分のみ追跡 — `egs5-operator.md`参照）
- `docs/plan_egs5_crosscheck.md`のステータス行・該当Phaseのチェックボックスを
  結果に合わせて更新する

## 想定される差異要因（不一致調査の初期チェックリスト）

`docs/plan_egs5_crosscheck.md`の表を参照。要点:

| 要因 | 方向性 | 対処 |
|---|---|---|
| カーマ近似 vs 電子輸送 | 界面近傍で過小/過大 | 診断領域では電子飛程<1mmなので通常無視可 |
| 蛍光X線の無視（ChatCarlo未実装） | 重元素近傍で過大・遠方で過小 | 既知の未実装。差が出れば実装優先度の根拠になる |
| 断面積ライブラリ差 | 全域で±1%程度 | 許容基準に織り込む |
| 組成規格差（ICRP vs ICRU-44） | 骨等で数% | 材料ごとの許容誤差（`tests/test_materials.py`の`_LOOSE`方式） |
| スペクトル離散化・線源の一様近似 | 系統差 | 同一離散スペクトルを両者に渡し線源起因の差を排除 |
| IBOUND不一致（コンプトン断面積モデル） | 全域で数%、水60keVで相対4.7% | 必ずIBOUND=1で揃える（`egs5-operator.md`） |

## 原則

- コードの数値差を疑う前に、まず物理モデル・比較条件が揃っているかを疑う
- 許容基準は実行前に固定し、結果を見てから緩めない
- EGS5の汎用オペレーターAI化（scene.yaml全機能の自動EGS5変換）はスコープ外
  （`docs/plan_egs5_crosscheck.md`の「やらないこと」）
