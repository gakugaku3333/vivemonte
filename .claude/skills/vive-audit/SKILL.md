# vive-audit — 独立監査の依頼

シミュレーションの要所で、`.claude/agents/vive-auditor.md` に定義された
独立監査官（氷室 律ペルソナ）に成果物を監査させるスキル。
**作業した本人（メインの会話）が自己申告で「問題ない」と言うのを防ぐ**のが
目的なので、監査は必ず別エージェントに委任し、メイン側でチェックリストを
なぞって代用しない。

## 引数

- 監査対象: scene.yaml のパス（必須）。加えてあれば run の出力ログ
  （テキストで渡す）、dose.npz のパス、ユーザーに出す予定の報告文案、
  `runs/<シーン名>/requirements.md`（vive-interviewが書き出したヒアリング要件記録）。
- 未指定なら、この会話で直近に扱ったシーン・結果を対象にする。
  `runs/<シーン名>/requirements.md` が存在するならステージAで必ず渡す
  （scene.yamlの物理的な正しさだけでなく、ユーザーが実際に頼んだ内容との
  一致を監査官が検証できるのはこのファイルがある場合のみ）。

## 手順

1. **ステージを判定する**。渡された成果物から監査ステージを決める:
   - scene.yaml のみ → ステージA（シーン監査）
   - run 出力 / dose.npz あり → ステージB（実行結果監査）
   - 報告文案あり → ステージC（報告文監査）
   複数該当なら該当ステージを全て依頼文に列挙する。

2. **Agentツールで監査官を起動する**。`subagent_type: "vive-auditor"` を指定
   （このセッションで型が未登録なら `general-purpose` で起動し、依頼文の
   冒頭で「まず `.claude/agents/vive-auditor.md` を読み、その人格と手順に
   完全に従うこと」と指示する）。依頼文には必ず含める:
   - リポジトリルートの絶対パスと、監査対象ファイルのパス
   - `runs/<シーン名>/requirements.md` があればそのパス（ステージAでは必須級。
     ないなら「要件記録なし」と明示し、ユーザー意図との一致検証が
     省略される旨を監査官に伝える）
   - run 出力ログの全文（コンソール出力はエージェントから見えないため）
   - 監査ステージ（A/B/C）
   - 「実装・修正は禁止、監査報告書の書式で返答」という念押し

3. **監査報告書をそのまま提示する**。要約して丸めず、判定・所見表・
   独立検証・未確認事項をユーザーに見せる。監査官の判定を書き換えない。

4. **差し戻しの場合**: 所見に沿って修正した後、**同じ観点で再監査**を依頼する
   （SendMessageで同じ監査官エージェントに続きを依頼すると文脈が保たれる）。
   合格が出るまで結果をユーザーへの最終報告に使わない。

## vive-check との連携

vive-check の関門で以下のタイミングに挟む（ユーザーが監査を明示的に
断った場合は省略可）:

- 関門1の承認前: ステージA監査 → 所見を添えて AskUserQuestion
- 関門3の直後（結果提示前）: ステージB監査
- 最終報告の直前: 数値を含む報告文案に対してステージC監査

## 注意

- 監査官が「差し戻し」でも最終判断はユーザーにある。判定を隠さず提示した上で
  「このまま進めるか」を確認するのは可。
- 監査官の指摘が誤っている（一次データと照合して反証できる）場合は、
  反証の根拠を添えて再監査に回す。水掛け論はユーザーに裁定を仰ぐ。
